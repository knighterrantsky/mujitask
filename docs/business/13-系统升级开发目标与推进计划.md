# 系统升级开发目标与推进计划

更新时间：`2026-04-12`

## 1. 文档目的

本文用于把“系统升级收敛方案”进一步落成可执行的开发计划。

本文回答 4 个问题：

- 这次升级的开发目标是什么
- 这次升级应该通过什么形式推进
- 实际的落地顺序是什么
- 每个阶段要交付什么、做到什么程度才算完成

本文默认承接上一份方案文档：

- [12-系统架构升级方案.md](./12-系统架构升级方案.md)

## 2. 本次升级的项目定义

本次升级不定义为“重写业务平台”，而定义为：

**一个兼容式执行控制升级专项。**

它的核心不是改业务抓取逻辑，而是先把当前系统从“同步直跑脚本”升级成“可排队、可加锁、可追踪、可持久化”的执行体系。

本次升级默认保留这些稳定边界：

- 保留现有 CLI 入口
- 保留现有 workflow / flow 业务实现
- 保留现有 runtime 文件协议
- 保留现有 OpenClaw `__OPENCLAW_RESULT__` 协议

## 3. 开发目标

## 3.1 一级目标

### 目标 A：保住当前稳定业务边界

升级过程中不能破坏当前真实稳定边界：

- `automation-business-scaffold-run`
- OpenClaw skill step 脚本
- `runtime/cli_runs` 与 `runtime/artifacts`
- `__OPENCLAW_RESULT__`

验收标准：

- 老的 CLI 仍然可独立运行
- 老的 skill 仍然可以按现有方式执行
- 新系统上线前后，业务结果口径保持一致

### 目标 B：解决单 profile 资源竞争

这是本轮升级的最高优先级问题。

必须做到：

- 同一浏览器 profile 不会被多个任务同时占用
- 任务支持排队
- 执行中任务有租约和心跳
- 异常退出后资源可以自动回收

验收标准：

- 同一 `profile_ref` 下并发提交 2 个任务时，只有一个进入运行态
- 后一个任务处于可观察的排队态，而不是直接冲突失败

### 目标 C：把执行状态从“本地现场”升级成“控制面可读”

必须把以下状态沉淀为结构化数据：

- 请求是否已接收
- 当前是否排队
- 当前由哪个 worker 执行
- 当前 `run_id` 是什么
- 是否成功
- 如果失败，失败摘要是什么

验收标准：

- 不进入宿主机目录，也能通过控制面查询任务状态
- 一个执行单元至少能查到 `request -> execution -> run_id`

### 目标 D：把运行产物从“本地磁盘”升级成“对象存储可追踪”

第一阶段不改 step 内部写文件逻辑，但要做到：

- 本地 runtime 文件执行后可统一上传
- 上传后的对象有索引
- 能通过 `run_id` 找到对应对象

验收标准：

- 至少完成以下文件上传与索引：
  - `run.json`
  - `steps.json`
  - `signals.json`
  - `stdout.log`
  - `artifacts/<run_id>/...`

### 目标 E：为后续异步化保留兼容演进空间

第一阶段不强制把 OpenClaw 改成纯异步，但要预留好后续接口：

- `submit`
- `status`
- `result`

验收标准：

- 当前同步协议不破坏
- 新的异步查询协议有文档、有字段、有可实现路径

## 3.2 二级目标

这些目标有价值，但不作为第一阶段上线阻塞项：

- 商品快照入库
- 外部记录绑定
- 通知 outbox
- 达人链 / 视频链接入
- 多节点 worker
- 业务配置中心数据库化

## 3.3 明确非目标

本轮不做的事情必须明确写清：

- 不重写 TikTok / FastMoss / 飞书业务 flow
- 不先做五张飞书表的平台化建模
- 不把字段映射、skill 映射、通知模板先做成数据库平台
- 不先拆微服务
- 不先引入 Celery / Kafka / Airflow

## 4. 推荐推进形式

## 4.1 以“兼容改造专项”推进，不以“大平台项目”推进

推荐把这次升级定义为一个独立专项，目标是：

- 升级执行控制
- 不打断现有产出
- 保持新老链路可共存

不建议的推进方式：

- 一次性推翻现有 OpenClaw 链路
- 一次性改写所有业务 task
- 一次性引入完整元数据平台

## 4.2 采用“纵切 MVP + 特性开关”推进

推荐形式：

- 先打通一条最短闭环
- 用特性开关选择新旧执行链
- 逐步扩大覆盖范围

建议第一条纵切链路：

```text
submit task
-> queue
-> resource lease
-> subprocess 调现有 CLI
-> 读取 runtime
-> 上传 MinIO
-> 回写执行结果
```

这样做的好处是：

- 开发路径短
- 业务改动小
- 出问题时可以快速回退到老链路

## 4.3 采用“文档先行 + 契约冻结 + 试点上线”的推进节奏

推荐每一阶段都按这个顺序推进：

1. 先冻结接口和数据契约
2. 再实现最小代码
3. 再接一条真实业务试点
4. 最后再扩大覆盖范围

先要冻结的不是 SQL，而是下面 3 个契约：

- `submit` 请求契约
- `status/result` 查询契约
- `__OPENCLAW_RESULT__ v2` 兼容返回契约

## 4.4 采用“阶段评审制”推进

每个阶段都设置进入条件和退出条件，不满足退出条件不进入下一阶段。

建议节奏：

- Phase 0：方案冻结
- Phase 1：执行控制 MVP
- Phase 2：对象存储接入
- Phase 3：实体快照与绑定
- Phase 4：通知补偿
- Phase 5：达人链 / 视频链扩展

## 5. 实际落地顺序

## 5.1 先做文档和契约，不先做数据库大表

第一步必须先补齐这些文档：

- 开发目标与推进计划
- Phase 1 模块目录草案
- `submit/status/result` 协议草案
- `__OPENCLAW_RESULT__ v2` 兼容字段草案
- 第一阶段最小表结构草案

原因：

- 这次最容易失控的地方不是代码，而是边界不清
- 文档先冻结，后面实现和 review 会顺很多

## 5.2 先做执行控制 MVP，不先做实体平台

第一阶段优先顺序应该是：

1. `task_request`
2. `task_execution`
3. `resource_lease`
4. `artifact_object`
5. `executor_daemon`
6. `submit_task`
7. `artifact_sync`

这一阶段先不要引入：

- `entity_registry`
- `external_binding`
- `entity_snapshot`
- `notification_outbox`

原因：

- 执行控制是上线前提
- 实体与快照是效果增强，不是执行前提

## 5.3 先接一条真实业务，不先全面切换

推荐 Phase 1 的试点任务只选一个：

- 首选：`feishu_single_row_update`

原因：

- 它最能覆盖浏览器占用
- 它最能覆盖失败现场
- 它最能覆盖 artifact 存档
- 它最适合验证排队和租约

如果试点稳定，再逐步接入：

1. `fastmoss_keyword_candidate_discovery`
2. `feishu_seed_row_insert`
3. `feishu_pending_rows_scan`
4. `tiktok_product_link_cleanup`

## 5.4 先保持同步兼容，再引入异步查询

OpenClaw 当前协议仍然以同步模式为主，所以顺序必须是：

1. 先保留当前同步返回
2. 再增加对象存储字段
3. 再增加 `status/result` 查询能力
4. 最后再评估是否切纯异步

不建议一上来就把 OpenClaw 改成“只 submit，不等待结果”。

## 6. 分阶段计划

## 6.1 Phase 0：方案冻结

目标：

- 把边界和契约冻结

交付物：

- 本文档
- `submit/status/result` 协议草案
- `__OPENCLAW_RESULT__ v2` 草案
- Phase 1 表结构草案
- Phase 1 模块目录草案

退出条件：

- 全部关键字段命名确定
- 新老链路切换方式确定
- Pilot workflow 确定为 `feishu_single_row_update`

## 6.2 Phase 1：执行控制 MVP

目标：

- 引入最小控制面
- 解决单 profile 冲突
- 保持业务 CLI 原样运行

建议模块：

- `submit_task.py`
- `executor_daemon.py`
- `queue_repo.py`
- `lease_repo.py`
- `execution_repo.py`

建议最小表：

- `task_request`
- `task_execution`
- `resource_lease`

执行方式：

- `executor_daemon` 轮询待执行任务
- 获取资源租约
- 用 `subprocess` 调现有 CLI
- 回写执行状态

退出条件：

- 同一 profile 下任务可排队
- 执行中断后租约可回收
- 可以通过数据库查到 `request_id -> execution_id -> run_id`

## 6.3 Phase 2：对象存储接入

目标：

- 将本地 runtime 文件同步到 MinIO

建议模块：

- `artifact_store.py`
- `minio_store.py`
- `artifact_sync.py`

建议表：

- `artifact_object`

执行方式：

- CLI 继续写本地 runtime
- 任务完成后由 executor 统一上传
- 记录对象 key、大小、etag、类型

退出条件：

- 单次 run 的核心文件全部可上传
- 可以通过 `run_id` 查询到对象索引
- 本地文件仍可作为兜底排障来源

## 6.4 Phase 3：实体快照与绑定

目标：

- 对商品链建立统一事实快照

建议模块：

- `entity_service.py`
- `binding_service.py`
- `snapshot_service.py`
- `diff_service.py`

建议表：

- `entity_registry`
- `external_binding`
- `entity_snapshot`

本阶段只先支持：

- `entity_type = product`

退出条件：

- 能基于 `SKU-ID / 商品ID / URL 提取 ID` 建立稳定商品实体
- 能沉淀一次执行后的商品事实快照

## 6.5 Phase 4：通知补偿

目标：

- 把执行结果通知从“直接发送”升级成“可补偿发送”

建议模块：

- `alert_service.py`
- `outbox_dispatcher.py`
- `openclaw_channel_adapter.py`

建议表：

- `notification_outbox`

退出条件：

- 成功 / 失败通知可追踪
- 通知失败可重试

## 6.6 Phase 5：达人链 / 视频链扩展

目标：

- 在不重做平台的前提下，复用同一套执行控制和对象存储体系

本阶段新增内容：

- `entity_type = creator`
- `entity_type = video`
- 新的 skill 配置与映射

退出条件：

- 新链路可以接入相同 executor / DB / MinIO 体系
- 不需要另起一套控制平台

## 7. 计划交付物清单

建议把交付物拆成 4 类。

### 7.1 文档交付物

- 系统升级收敛方案
- 开发目标与推进计划
- 协议文档
- 表结构文档
- Pilot 试点运行说明

### 7.2 代码交付物

- executor daemon
- queue / lease / execution repository
- MinIO 上传同步模块
- 开关式的新旧执行入口

### 7.3 数据交付物

- Alembic 迁移
- Phase 1 核心表
- Phase 2 artifact 索引表

### 7.4 运维交付物

- 本地或测试环境 Postgres
- MinIO bucket 与 lifecycle 配置
- executor 运行脚本
- 回滚开关与回退说明

## 8. 每阶段验收方式

每个阶段建议都做 3 层验收。

### 8.1 契约验收

- 请求字段是否符合文档
- 响应字段是否符合文档
- 新老协议是否兼容

### 8.2 行为验收

- 同一 profile 是否正确排队
- 失败后是否释放租约
- 对象是否成功上传
- 状态是否可回查

### 8.3 业务验收

- 试点任务结果是否与旧链路一致
- 飞书最终写入结果是否一致
- 失败排障路径是否更清楚

## 9. 推荐实施节奏

如果按照最稳妥的节奏推进，建议按“阶段完成”而不是按自然周承诺。

推荐顺序：

1. 先交付 Phase 0 文档与契约
2. 再做 Phase 1 MVP
3. 再做一条 Pilot 试点
4. Pilot 稳定后再补 MinIO
5. MinIO 稳定后再做快照与绑定

也就是说：

**先解决执行控制，再解决产物持久化，再解决数据沉淀，最后再扩业务范围。**

## 10. 最终执行建议

如果现在正式启动，我建议按下面的方式推进：

### 启动方式

- 以独立专项推进
- 以 worktree / 独立分支承接
- 以文档先行和 Phase 评审制管理

### 第一批实际任务

- 写 `submit/status/result` 协议草案
- 写 Phase 1 的 4 张核心表 DDL
- 写 executor 模块目录草案
- 选 `feishu_single_row_update` 做 pilot

### 第一批不做的任务

- 不先接达人链
- 不先做配置中心
- 不先做五表平台化
- 不先改业务 flow 内部结构

一句话总结：

**这次升级最正确的推进方式，不是“边想边重构业务”，而是“先冻结契约，再做执行控制 MVP，再用一条真实业务试点把新链路跑通”。**
