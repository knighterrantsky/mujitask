# TK事实数据库升级影响评估

更新时间：`2026-04-21`

## 1. 升级结论

本次升级从旧的通用 `entity_registry / external_binding / entity_snapshot` 模型，切换为明确的 TikTok 业务主体模型。

旧 entity 层不再兼容：

- 不再创建 `entity_registry`。
- 不再创建 `external_binding`。
- 不再创建 `entity_snapshot`。
- 不再写入 product / influencer 的通用 entity snapshot。
- 不再在结果 payload 中输出 `entities / entity_bindings / entity_snapshots`。

新的事实库按业务语义拆分：

- 主体层：商品、SKU、店铺、达人、视频。
- 素材层：主图、侧边栏图、头像、视频封面、页面截图。
- 关系层：商品-店铺、达人-商品、达人-视频、视频-商品、店铺-达人。
- Raw 审计层：原始接口响应和响应命中的主体链接。
- 快照层：表先预建，但现阶段不由现有流程写入。

## 2. 旧 entity 层移除影响

旧 entity 层原先承担两件事：

1. 商品刷新流程结束后，把商品结果写入 `entity_snapshot`。
2. 达人池流程结束后，把达人状态写入 `entity_snapshot`，并在后续合并达人状态时读取 latest snapshot。

本次调整后：

- 商品刷新流程改为写入 `tk_products / tk_shops / tk_product_shop_relations / tk_media_assets`。
- 达人池流程改为写入 `tk_creators / tk_products / tk_shops / tk_creator_product_relations / tk_shop_creator_relations / tk_media_assets`。
- 达人是否已处理过某个来源商品，优先从飞书目标表记录和 `tk_creator_product_relations` 判断。
- 旧 entity snapshot 历史不迁移、不回填。

## 3. 保留不变的现有业务

以下执行控制和队列表不受本次升级影响：

- `task_request`
- `task_execution`
- `resource_lease`
- `notification_outbox`
- `artifact_object`
- `influencer_pool_product_job`
- `influencer_pool_author_job`

以下业务行为保持不变：

- 商品刷新流程继续采集并写回现有飞书竞品字段。
- 达人池流程继续采集并写回现有飞书达人池字段。
- 飞书业务表映射和数据库到飞书的新写回逻辑本阶段不开发。

## 4. 新增表

主体主档：

- `tk_products`
- `tk_product_skus`
- `tk_shops`
- `tk_creators`
- `tk_videos`

素材附属：

- `tk_media_assets`
- `tk_entity_media_assets`

主体关系：

- `tk_product_shop_relations`
- `tk_creator_product_relations`
- `tk_creator_video_relations`
- `tk_video_product_relations`
- `tk_shop_creator_relations`

Raw 审计：

- `tk_raw_api_responses`
- `tk_raw_entity_links`

预建快照/窗口：

- `tk_product_daily_metrics`
- `tk_product_window_latest`
- `tk_product_window_observations`
- `tk_product_distribution_window_latest`
- `tk_product_distribution_window_observations`
- `tk_product_sku_window_latest`
- `tk_product_sku_window_observations`
- `tk_video_product_window_performance`
- `tk_creator_product_window_performance`

## 5. 本阶段不做

- 不写销量快照。
- 不写窗口观察值。
- 不写节日复盘快照。
- 不做旧 `entity_snapshot` 数据迁移。
- 不做数据库到飞书业务表的新映射。
- 不做数据库到飞书的新写回。

后续快照写入策略应作为单独应用层接口开发，由业务接口决定是否把采集结果写入快照表。
