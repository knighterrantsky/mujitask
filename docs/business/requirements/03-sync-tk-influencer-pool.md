# 竞品到达人池同步需求

更新时间：`2026-04-23`

- 入口任务：`sync_tk_influencer_pool`
- 触发方式：每天一次定时任务
- 来源表：`TK竞品收集`
- 目标表：`TK达人池`
- 共用口径：表结构、自动维护字段、非自动维护字段以 [../01-需求文档.md](../01-需求文档.md) 为准。

## 1. 流程定位

客户希望 `OpenClaw` 每天定时执行一次“竞品到达人池”的同步任务，从 `TK竞品收集` 中查找尚未执行达人查找的商品，把满足条件的达人沉淀到 `TK达人池`，并在竞品表中记录该商品是否已经完成达人查找。

该流程只负责从竞品商品扩展达人画像与关联商品沉淀，不负责新增竞品商品，也不负责刷新竞品商品本身的自动维护字段。

## 2. 客户需求

系统需要满足：

1. 每天自动扫描 `TK竞品收集` 中尚未完成达人查找的商品。
2. 基于商品对应的达人数据筛选目标达人，并把结果新增或更新到 `TK达人池`。
3. 在 `TK竞品收集` 中维护 `达人查找状态`，用于标记该商品是否已完成达人查找。
4. 对已经完成达人查找的商品默认不重复处理；如需重跑，由运营人工重置状态。
5. 对达人池只维护本期已明确的达人画像和关联商品字段，支持后续达人运营和业务分析。

## 3. 信息采集与回写逻辑

`TK达人池` 当前承担的是达人侧沉淀和后续运营分析主表角色。

在定时同步场景下，该表的业务逻辑如下：

1. 系统每天定时从 `TK竞品收集` 中扫描尚未完成达人查找的商品，作为达人池同步的来源数据。
2. 候选来源固定为 `达人查找状态` 为空、`待查找`、`失败重试`，以及上一轮异常残留为 `处理中` 的商品记录；`已完成` 的记录默认跳过。
3. 如果来源商品已被标记为不可访问，则该商品不再进入达人查找流程。
4. 系统基于来源商品对应的达人数据筛选目标达人，并把满足条件的达人新增或更新到 `TK达人池`。
5. `TK达人池` 保持一人一行，系统固定按 `达人ID` 做新增或更新。
6. 达人筛选通过后，达人画像信息和关联商品信息一起沉淀到 `TK达人池`；其中：
   - `带货商品图` 直接使用来源 `TK竞品收集` 商品行中的商品图片；
   - `关联商品销量` 取该达人在该来源商品上的对应销量数据，不取达人整体历史总销量；
   - `关联节日` 直接取来源 `TK竞品收集` 商品行中的 `节日` 值；
   - `合作店铺` 取该达人当前对应的合作店铺数据，并写入 `TK达人池` 的 `合作店铺` 字段。
7. 当同一个达人命中多个商品时，系统在原有达人行上继续补充和累加本期约定的达人画像与关联商品信息，而不是拆成多行；其中：
   - 商品图片按命中的不同来源商品继续追加；
   - 关联商品销量按命中商品上的对应达人销量继续累加；
   - 关联节日按并集方式合并，相同节日不重复写入；
   - 合作店铺按并集方式合并，相同店铺不重复写入；仅写入当前飞书字段中已存在的店铺选项。
8. 达人联系方式属于本期自动维护字段之一；当系统能拿到有效联系方式时写入或补齐，当本次拿不到稳定值时允许留空，且不应用空值覆盖已有内容。
9. `粉丝数`、`带货视频 GMV`、`带货直播 GMV` 在数据库和实体快照中保留实际数字；写入飞书表做最终展示时，数值大于等于 `10000` 的统一显示为 `W` 单位，例如 `10000 -> 1W`、`1230000 -> 123W`，小于 `10000` 的直接显示原始数字。
10. `合作商品数` 不作为本期更新字段，不参与达人池自动写回。
11. 本期只自动维护已明确约定的达人池字段；历史沉淀字段、人工运营字段和公式字段不纳入自动维护范围。
12. `记录时间` 只表示该达人行最近一次成功更新的日期；当本次确实完成达人池写回时才刷新。
13. `TK竞品收集` 中的 `达人查找状态` 用于标记该商品是否已经完成达人查找；已完成记录默认不重复处理，只有人工重置后才重新进入流程。
14. 如果本次达人查找未能完整完成，则来源商品保持待重试状态，等待后续定时任务继续处理。

## 4. 最终交付形式

最终交付形式是 OpenClaw skills。

skills 需要支持在 `OpenClaw` 中配置每天执行一次的定时任务，把 `TK竞品收集` 中尚未完成达人查找的商品继续同步到 `TK达人池`。

在该场景下，skills 需要完成以下动作：

1. 从 `TK竞品收集` 中扫描 `达人查找状态` 为 `待查找`、`失败重试`、空值，以及上一轮异常残留为 `处理中` 的记录。
2. 跳过 `商品状态=已下架/区域不可售` 的记录，以及 `达人查找状态=已完成` 的记录。
3. 对进入处理的记录先写 `达人查找状态=处理中`。
4. 从商品来源中读取达人候选，只把满足约定筛选条件的达人写入 `TK达人池`。
5. 以 `达人ID` 作为唯一主键，对 `TK达人池` 做新增或更新。
6. 对已存在达人按单行累加规则更新商品图、节日、关联商品销量和合作店铺；`合作商品数` 不作为更新字段。
7. `达人联系方式` 拿不到稳定值时允许留空，不阻塞其他字段写回。
8. 如果该商品达人列表处理完成，则回写 `达人查找状态=已完成`；如果中途失败，则回写 `达人查找状态=失败重试`。
9. 每次成功写回达人池字段时同步刷新达人池中的 `记录时间`；竞品表中的 `记录日期` 不表示达人查找完成时间。

## 5. 验收口径

1. 当 `TK竞品收集` 中某个商品命中一个新达人，且该达人满足约定筛选条件时，系统能在 `TK达人池` 中成功新建达人行。
2. 当商品命中的达人不满足筛选条件时，系统会直接跳过该达人，不写入 `TK达人池`。
3. 当同一个达人命中第二个不同商品时，系统仍保持 `TK达人池` 一人一行，并在原行上追加商品图、累加 `关联商品销量`、合并 `关联节日` 和 `合作店铺`；`合作商品数` 不作为更新字段。
4. 当第二个商品对应的 `节日` 与达人池中已有节日重复时，系统会跳过重复节日，不重复写入。
5. `合作店铺` 只会写入当前飞书字段中已存在的店铺多选项；来源数据返回但飞书字段中不存在的店铺名默认跳过。
6. 当 `达人联系方式` 取不到稳定正文时，系统仍能完成达人入池和其他字段更新。
7. 同一达人被重复刷新时，`记录时间` 会覆盖为最新同步日期。
8. `TK竞品收集` 中新增的 `达人查找状态` 字段固定为单选字段，状态值只有 `待查找 / 处理中 / 已完成 / 失败重试`。
9. 新增竞品记录在 `达人查找状态` 为空时，下一次定时任务会把它视为 `待查找` 并进入达人查找流程。
10. 每天定时任务只扫描 `达人查找状态` 为 `待查找`、`失败重试`、空值，以及上一轮异常残留为 `处理中` 的记录；`已完成` 的记录默认跳过。
11. 当某条竞品记录开始处理时，状态会先从空值或待处理态变为 `处理中`。
12. 当该商品达人列表扫描完成且所有满足条件的达人都已成功写入达人池后，该记录会被标记为 `已完成`。
13. 当该商品最终筛选出 `0` 个满足条件的达人时，该记录仍会被标记为 `已完成`。
14. 当达人池写入过程部分失败、或商品无法解析到可用的 FastMoss 商品详情入口时，该记录会被标记为 `失败重试`，并在后续日任务继续重试。
15. 当任务异常中断导致记录残留为 `处理中` 时，下一次日任务会重新拾起该记录，不会永久卡死。
16. 如果运营人工把 `达人查找状态` 改回 `待查找`，该记录会重新进入达人查找流程。

## 6. 变更影响边界

本流程文档可独立变更以下内容：

1. `达人查找状态` 的待处理筛选和状态流转规则。
2. 达人筛选条件。
3. 达人画像字段、关联商品字段的追加、累加和合并规则。
4. 达人池写入失败、重试和异常恢复规则。
5. 本流程的交付动作和验收口径。

以下变更需要同步更新主需求文档：

1. `TK竞品收集` 或 `TK达人池` 表结构变化。
2. 自动维护字段或非自动维护字段的共用口径变化。
3. 新增或废弃正式业务流程。
4. 本流程变更会影响竞品表定时刷新或关键词新增竞品流程。
