# 竞品采集需求

更新时间：`2026-07-23`

- 入口任务：`refresh_current_competitor_table`
- 触发方式：每天一次定时任务
- 业务主表：`TK竞品收集`
- 共用口径：表结构、自动维护字段、非自动维护字段以 [../business-requirements.md](../business-requirements.md) 为准。

## 1. 流程定位

客户会先在飞书多维表中维护需要持续跟踪的商品记录，随后由 `OpenClaw` 通过每天一次的定时任务自动更新竞品信息，并写回 `TK竞品收集`。

该流程只负责维护已有竞品记录的商品侧和 FastMoss 侧数据，不负责新增关键词候选商品，也不负责达人池同步。

## 2. 客户需求

系统需要满足：

1. 每天自动触发一次竞品数据更新。
2. 对飞书表中的商品记录做标准化、去重和待更新筛选。
3. 对待更新商品抓取最新业务数据并写回 `TK竞品收集`。
4. 如果商品明确不可访问、已下架或区域不可售，则在表中标记对应状态并停止后续抓取。

## 3. 信息采集与回写逻辑

`TK竞品收集` 当前承担的是商品侧竞品运营主表角色。

在定时任务场景下，该表的业务逻辑如下：

1. 系统每天定时扫描 `TK竞品收集`，只处理仍需补齐自动维护字段的商品记录。
2. 待更新判断只基于当前约定的 13 个自动维护字段；只有这些字段中至少一个为空的记录才进入采集队列。
3. `商品状态`、`达人查找状态`、`备注`、截图字段和其他客户自行维护字段不参与待更新判断，也不会因为为空而创建采集 job。
4. 在正式抓取前，系统先识别并清理空行，再对商品记录做标准化和去重，避免重复商品反复进入后续流程。
5. 这里的空行是指该行所有字段都没有有效值；如果只是 `产品链接` 为空但其他字段仍有值，则不视为空行。
6. 去重按标准化后的 `产品链接` 进行，遇到重复记录时保留首条，删除后续重复行。
7. 当一条记录存在 `产品链接` 时，系统优先以 `产品链接` 作为商品入口；当缺少 `产品链接` 但存在 `SKU-ID` 时，系统可以通过 `SKU-ID` 继续定位商品。
8. 如果人工修改了 `产品链接` 或 `SKU-ID`，需要同时人工清空 `商品状态`，让该商品重新进入抓取流程。
9. 每条待采集飞书记录只创建一个行级采集 job；该 job 内部串行完成 TikTok 商品信息拉取、必要浏览器兜底、媒体同步、FastMoss 商品信息拉取、数据库入库和飞书写回。
10. 商品侧基础信息来自商品本身的页面信息；FastMoss 侧价格、佣金率和销量信息来自该商品对应的 FastMoss 商品详情数据。
11. 如果商品明确不可访问、已下架或区域不可售，则系统在该行写入 `商品状态`，并停止该商品后续抓取。
12. 如果商品可正常访问，则系统补齐本期约定的商品自动维护字段，并将结果写回 `TK竞品收集`。
13. 写回时只补齐当前为空的自动维护字段，不覆盖已有值，也不把 `关键词`、`备注`、阶段分析字段等客户自行维护内容纳入自动维护范围。
14. 如果本次只需要补齐商品侧信息，而 FastMoss 侧业务数据已经完整，则系统不重复进入 FastMoss。
15. `记录日期` 只表示该商品信息最近一次成功写回的日期；只有本次确实产生写回时才刷新。

## 4. 最终交付形式

最终交付形式是 OpenClaw skills。

skills 需要支持在 OpenClaw 中配置定时任务，并在定时任务触发后完成以下动作：

1. 读取 `TK竞品收集` 中的商品记录。
2. 在链接预处理阶段删除空行。
3. 对 `产品链接` 做格式化处理，并对重复链接做去重。
4. 获取待更新的数据；自动维护字段已完整的行直接跳过。
5. 按第 3 节约定的 `TK竞品收集` 定时任务逻辑逐条抓取并更新飞书表。
6. 已被标记为 `已下架/区域不可售` 的商品默认直接跳过，除非人工先清空 `商品状态`。

## 5. 验收口径

1. 客户可以继续使用当前飞书多维表；`TK竞品收集` 当前表结构中包含 `Fastmoss价格`、`佣金率`、`商品状态` 和 `达人查找状态` 字段。
2. 文档中的自动维护字段固定为 `产品链接 / SKU-ID / 图片 / 标题 / 节日 / 卖家 / 价格 / Fastmoss价格 / 佣金率 / 昨日销量 / 近7天销量 / 近90天销量 / 记录日期`，`商品状态` 不属于这 13 个自动维护字段。
3. 待更新判断只基于自动维护字段，`商品状态`、`达人查找状态`、`备注`、`前台截图`、`Fastmoss截图` 和其他客户自行维护字段不参与待更新判断。
4. 进入采集队列的记录数必须等于“13 个自动维护字段至少一个为空，且未因空行、重复、已下架/区域不可售、无有效商品入口被跳过”的记录数。
5. 客户在飞书表中新增 `产品链接` 后，OpenClaw 能按天定时更新竞品信息。
6. 定时更新前，skills 能在链接预处理阶段识别空行并删除整行。
7. 这里的空行定义为 cleanup 读取到的原始字段中所有字段都没有有效值；如果只有 `产品链接` 为空但其他字段仍有值，则不属于空行。
8. 定时更新前，skills 能对 `产品链接` 做格式化并对重复链接去重。
9. 重复 `产品链接` 去重时保留首条，删除后续重复行。
10. 当有 `产品链接` 时，系统能基于 `产品链接` 抓取数据。
11. 当没有 `产品链接` 但有 `SKU-ID` 时，系统能基于 `SKU-ID` 拼接商品链接并抓取数据。
12. 如果 TikTok request 阶段已经明确识别商品不可访问、已下架、不存在或区域不可售，系统会直接把 `商品状态` 标记为 `已下架/区域不可售`，并停止该行后续浏览器兜底、媒体同步和 FastMoss 补齐。
13. 被标记为 `已下架/区域不可售` 的商品，后续默认直接跳过；如果人工改了 `SKU-ID` 或 `产品链接`，需要同时人工清空 `商品状态` 才能重新抓取。
14. 只有在商品页可正常访问且仍缺 FastMoss 侧业务数据时，系统才会进入 FastMoss 补齐对应信息。
15. 如果当前仅缺商品侧字段而 FastMoss 侧业务数据已完整，系统不会重复进入 FastMoss。
16. 采集完成后，系统能把对应自动维护字段写回飞书多维表，并更新 `记录日期`。
17. `佣金率` 取 FastMoss 商品详情基础数据中的商品佣金率并按百分比文本写回，例如 `10%`；FastMoss 明确返回 `-` 时原样写入 `-`，接口未返回该字段时不伪造百分比值。
18. Runtime 结果必须证明每条待采集记录只创建一个行级主 job，不允许按同一记录拆成 TikTok request job 和 FastMoss job 并行执行。
19. TikTok 浏览器兜底必须有 runtime 证据：已实际发起 request、记录 HTTP/错误/风控信号，并且 fallback reason 是明确的风控、登录、验证码、访问受限或缺少商品详情脚本；明确不可售不进入 fallback，网络失败、超时、5xx 或 429 先走重试。

## 6. 变更影响边界

本流程文档可独立变更以下内容：

1. 定时刷新任务的待处理筛选规则。
2. 空行清理、链接标准化和去重规则。
3. 商品不可访问、下架或区域不可售的判断和状态写回规则。
4. 本流程的交付动作和验收口径。

以下变更需要同步更新主需求文档：

1. `TK竞品收集` 表结构变化。
2. 自动维护字段或非自动维护字段的共用口径变化。
3. 新增或废弃正式业务流程。
4. 本流程变更会影响关键词搜索竞品写入或达人池同步流程。
