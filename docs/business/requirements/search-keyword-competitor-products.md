# 关键词搜索竞品写入需求

更新时间：`2026-07-23`

- 入口任务：`search_keyword_competitor_products`
- 触发方式：OpenClaw 对话输入
- 业务主表：`TK竞品收集`
- 共用口径：表结构、自动维护字段、非自动维护字段以 [../business-requirements.md](../business-requirements.md) 为准。

## 1. 流程定位

客户会在 `OpenClaw` 中输入自然语言需求，例如让系统按关键词搜索 `TK` 商品，并把符合条件的结果写入 `TK竞品收集`。

该流程只负责按关键词发现和新增竞品种子记录，并复用竞品表统一采集与回写逻辑补齐自动维护字段；已存在商品默认跳过。

## 2. 客户需求

系统需要满足：

1. 能识别用户输入中的关键词和业务筛选条件。
2. 基于 FastMoss 搜索 `TK` 商品并返回符合条件的候选结果。
3. 将新增商品写入 `TK竞品收集`，并补齐约定的自动维护字段。
4. 已存在的商品默认跳过，不重复新增。
5. 对通过搜索新增的商品，保留来源信息，便于后续追溯。

## 3. 信息采集与回写逻辑

在关键词搜索场景下，`TK竞品收集` 的业务逻辑如下：

1. 用户通过 `OpenClaw` 输入自然语言需求，系统识别关键词和业务筛选条件后，基于 FastMoss 搜索 `TK` 商品。
2. 搜索结果固定按 `近7天销量` 倒序排序，并按页持续获取候选商品。
3. 系统只保留满足用户业务阈值的候选商品，并先判断这些商品是否已存在于 `TK竞品收集`。
4. 已存在商品直接跳过，不重复新增，也不重写现有记录。
5. 不存在商品先写入 `SKU-ID` 和对应的标准化 `产品链接`，再复用本表的统一采集与回写逻辑补齐包括 `佣金率` 在内的其他自动维护字段。
6. 通过搜索新增的商品，需要在 `备注` 中保留来源信息，格式为 `通过搜索关键字：{关键词}`，便于后续追溯。
7. 如果新商品在后续详情抓取阶段被判定为不可访问，则系统保留该商品已写入的 `SKU-ID`、`产品链接` 和 `备注`，并额外写入 `商品状态=已下架/区域不可售`。

候选数量默认最多 `20` 条；当运行参数 `max_candidates=0` 时，表示不按候选数截断，系统会抓取所有满足条件的商品，直到分页、FastMoss total、空页、无新商品或销量阈值停止条件触发。

关键词搜索存在 `7 日销量阈值` 且 FastMoss 搜索明确按 `近7天销量` 倒序返回时，系统按整页判断是否继续翻页：如果当前页所有可解析商品的 `近7天销量` 最大值仍低于阈值，后续页默认视为不再可能满足阈值，停止继续翻页。该停止条件不会因为 `max_candidates=0` 被关闭；`max_candidates=0` 只取消候选数量上限，不取消分页、total、空页、无新商品和销量阈值等安全停止条件。

安全验证码属于异常风控处理，不属于正常采集固定步骤。正常 FastMoss 搜索、正常 TikTok request 和正常 TikTok 商品页浏览器采集，不应额外等待验证码图片或滑块元素。

当 FastMoss 搜索接口返回 `MSG_SAFE_0001` 时，系统先刷新登录态并重试原始搜索请求；若仍触发风控，才进入 FastMoss 浏览器风控解除流程。该流程只能验证原始搜索接口 `/api/goods/V2/search` 是否不再返回 `MSG_SAFE_0001`，不能用打开任意商品详情页作为搜索风控解除成功依据。

搜索后的详情补齐阶段如果 `fastmoss_product_fetch` 调用商品详情 API，例如 `/api/goods/v3/base`，返回 `MSG_SAFE_0001`，也必须进入同一 FastMoss provider 级 browser security resolve。handler 返回 `fallback_required` 和脱敏 `verification_request`，browser worker 只验证原始失败的 FastMoss API 请求不再返回 `MSG_SAFE_0001`，成功后持久化 FastMoss cookies 到 `fastmoss_session_cookie_cache`，再由行级主 job 重试原 FastMoss handler 一次。FastMoss cookies，不是 API token；cookie value 不得进入 summary/log/result。

FastMoss 账号会话恢复属于平台策略。若缓存 cookie 已标记 `last_auth_failed_at`，不得继续复用；若明确 auth 失效，平台层重新登录并持久化新 cookie。若刷新后仍 auth 失败，任务应返回 `fastmoss_session_conflict_or_external_login`，表示可能存在单点登录或外部登录冲突，需要账号使用环境收敛后再跑。

当 TikTok 商品详情采集触发安全验证滑块时，系统只在已识别风控 popup 后处理滑块。图片等待是“元素出现的最大等待”，不是固定等待；滑动后最多轮询 5 秒验证结果，弹窗消失后延迟 2 秒二次确认。如果二次确认时弹窗重新出现，或页面出现 `Unable to verify. Please try again.` 等失败态文本，本次滑块尝试视为失败并按重试策略继续。

TikTok 商品页验证码和 FastMoss 搜索验证码使用独立业务逻辑。TikTok 商品页默认沿用已通过真实 Roxy/TikTok 验证的 framework `SliderCaptchaResolver` match 策略，默认 `simple_target=false`；FastMoss 使用 FastMoss/Tencent selector profile。两类流程都必须在本地短期 artifact 目录保留截图、原始坐标、坐标换算、拖动距离、失败态文本和二次确认结果，便于排障；这些验证码诊断文件不得进入 MinIO，也不得作为跨进程业务输入。

## 4. 最终交付形式

最终交付形式是 OpenClaw skills。

skills 需要支持用户在 OpenClaw 对话中输入类似下面的指令：

`帮我查询/收集关键字为“east egg”的 7 日内销量大于 200 的 TK 商品数据`

在该场景下，skills 需要完成以下动作：

1. 从用户输入中识别 `关键词` 和 `7 日销量阈值`。
2. 通过 FastMoss 搜索竞品商品。
3. 固定按 `近7天销量` 倒序排序并按停止条件翻页获取结果。
4. 只保留 `近7天销量` 大于阈值的商品。
5. 判断候选商品是否已存在。
6. 已存在商品直接跳过，不更新该条数据。
7. 不存在商品先写入 `SKU-ID` 和按 `SKU-ID` 拼接并标准化后的 `产品链接`。
8. 再按第 3 节约定的 `TK竞品收集` 关键词搜索逻辑补全其他自动维护字段。
9. 对新写入商品在 `备注` 中标记 `通过搜索关键字：{关键词}`。
10. 如果新商品详情页明确不可访问，则把 `商品状态` 标记为 `已下架/区域不可售`，后续默认跳过。

## 5. 验收口径

1. 用户可以在 OpenClaw 中通过自然语言输入关键词检索需求。
2. 关键词搜索固定使用 FastMoss 数据源，固定按 `近7天销量` 倒序排序，并按分页、total、空页、无新商品、候选数或销量阈值停止条件结束。
3. 系统能根据关键词与 `7 日销量阈值` 获取 `0` 条或 `N` 条候选竞品商品。
4. 关键词搜索命中已存在商品时，会直接跳过，不更新、不重写该条数据。
5. 关键词搜索命中的新商品会先写入 `SKU-ID`，再补全包括 `佣金率` 在内的其他自动维护字段；`佣金率` 的来源、格式和缺失值口径与竞品采集流程一致。
6. 关键词搜索新增写入的商品会在 `备注` 中标记 `通过搜索关键字：{关键词}`。
7. 如果关键词搜索新增商品在详情抓取阶段被判定为不可访问，系统仍会保留该商品已有的 `SKU-ID`、`产品链接`、`备注`，并写入 `商品状态=已下架/区域不可售`。

## 6. 变更影响边界

本流程文档可独立变更以下内容：

1. 自然语言输入中关键词和筛选条件的识别口径。
2. FastMoss 搜索、排序、翻页和候选过滤规则。
3. 新增商品的判重规则和种子行写入规则。
4. 搜索来源备注格式。
5. 本流程的交付动作和验收口径。

以下变更需要同步更新主需求文档：

1. `TK竞品收集` 表结构变化。
2. 自动维护字段或非自动维护字段的共用口径变化。
3. 新增或废弃正式业务流程。
4. 本流程变更会影响竞品采集或达人池同步流程。
