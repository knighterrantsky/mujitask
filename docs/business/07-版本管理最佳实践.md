# 版本管理最佳实践

更新时间：`2026-04-10`

## 1. 文档目的

本文件用于约定当前工程中“需求版本、代码版本、文档版本”之间的关系，避免出现“代码已经变了，但版本号和文档口径对不上”的情况。

本文件不是业务需求文档，也不是设计文档，而是面向发布和协作的版本管理约定。

## 2. 当前工程推荐的版本分层

当前工程建议至少区分 3 层版本：

1. `需求版本`
2. `发布版本`
3. `文档版本`

推荐含义如下：

| 层级 | 作用 | 推荐格式 | 当前建议存放位置 |
| --- | --- | --- | --- |
| `需求版本` | 标识业务目标或规则集合的代际 | `v3.3` | [01-需求文档.md](./01-需求文档.md)、[02-设计文档.md](./02-设计文档.md) |
| `发布版本` | 标识实际交付的软件版本 | `3.3.3`、`3.3.4` | `pyproject.toml`、`src/automation_business_scaffold/__init__.py`、git tag、release note |
| `文档版本` | 标识文档内容对应的发布基线 | 推荐与发布版本一致，或改写为“适用发布版本” | 各业务文档头部或版本信息区 |

核心原则：

- `需求版本` 不等于 `发布版本`。
- 同一个 `需求版本` 下，可以存在多个连续的 `发布版本`。
- 只要代码行为发生了可交付变化，就不应该继续沿用旧的 `发布版本`。
- 文档应该说明“这份文档对应哪个发布版本”，而不是拿 `需求版本` 代替 `发布版本`。

## 3. 三类版本分别什么时候变

### 3.1 需求版本什么时候变

只有在业务目标、边界或规则集合发生一轮明显升级时，才提升 `需求版本`。

典型场景：

- 增加一条新的业务主链路
- 增加一类新的客户入口
- 调整关键字段体系或验收口径
- 改变整体产品能力边界

不建议因为一个小规则修正就把 `v3.3` 改成 `v3.4`。

### 3.2 发布版本什么时候变

只要本次改动准备进入发布包，就应该评估是否提升 `发布版本`。

建议沿用语义化版本的思路：

- `主版本`：不兼容的大变更
- `次版本`：兼容性内的新能力集合
- `修订版本`：兼容性内的规则补充、缺陷修复、行为修正、文档对齐

对于当前工程这类改动，通常可以这样理解：

- `3.3.3 -> 3.3.4`：在 `v3.3` 需求范围内新增一个兼容的小行为，比如 cleanup 增加空行删除
- `3.3.x -> 3.4.0`：出现一组新的业务能力，已经超出 `v3.3` 的需求边界

### 3.3 文档版本什么时候变

推荐优先采用下面两种方式之一：

1. `文档版本 = 发布版本`
2. 将 `文档版本` 改为 `适用发布版本`

当前工程如果继续保留“文档版本”字段，最佳实践是：

- 当文档描述的是某个可发布实现时，文档版本直接对齐该发布版本
- 当文档只是文字优化、不涉及发布行为变化时，可以只更新日期，不强制提升版本号

如果后续想进一步减少歧义，更推荐把文档里的：

- `文档版本：v3.3.3`

逐步改成：

- `适用发布版本：>= 3.3.4`
- 或 `适用发布版本：3.3.4`

这样更容易区分“文档自身修改”与“代码发布版本”。

## 4. 推荐的版本关联方式

### 4.1 一个需求版本对应多个发布版本

推荐关系如下：

- `需求版本` 表示业务代际
- `发布版本` 表示该代际下的一次具体交付

示例：

| 需求版本 | 发布版本 | 含义 |
| --- | --- | --- |
| `v3.3` | `3.3.3` | `v3.3` 范围内的一个已发布基线 |
| `v3.3` | `3.3.4` | 在 `v3.3` 范围内新增 cleanup 空行删除后的下一次发布 |
| `v3.4` | `3.4.0` | 新一轮需求代际的首次正式发布 |

### 4.2 每条重要规则最好有稳定 ID

推荐给关键规则或需求项增加稳定 ID，便于做追踪。

示例：

- `REQ-CLEANUP-EMPTY-ROW`
- `REQ-PENDING-ROWS-SCAN`
- `REQ-UNAVAILABLE-PRODUCT-STATUS`

有了 ID 之后，文档和发布说明可以更清晰：

- 需求文档描述 `REQ-CLEANUP-EMPTY-ROW`
- 设计文档说明该规则落在哪个 flow
- release note 写明“`3.3.4` 实现 `REQ-CLEANUP-EMPTY-ROW`”

## 5. 当前工程的推荐落地位置

建议按下面方式管理：

| 信息 | 文件或位置 |
| --- | --- |
| `发布版本` | [pyproject.toml](../../pyproject.toml)、[src/automation_business_scaffold/__init__.py](../../src/automation_business_scaffold/__init__.py) |
| `需求版本` | [01-需求文档.md](./01-需求文档.md) |
| `设计对应关系` | [02-设计文档.md](./02-设计文档.md) |
| `发布说明` | 建议新增 `CHANGELOG.md` 或 release note |
| `版本导航说明` | 本文件 |

建议再补一条简单约定：

- 仓库 tag 一律使用 `v3.3.4` 这类发布版本标签
- 不使用 `v3.3` 这种需求版本去打发布 tag

## 6. 发布前检查清单

每次准备发布时，建议按下面顺序检查：

1. 这次改动是否改变了外部可见行为。
2. 如果改变了行为，是否应该提升 `发布版本`。
3. 这次改动是否仍属于当前 `需求版本` 的边界。
4. 需求文档和设计文档是否已经更新到本次行为。
5. 文档中的版本口径是否与实际 `发布版本` 对齐。
6. 是否补充了 release note、变更记录或 changelog。
7. 是否打了对应的 git tag。

## 7. 针对本次 cleanup 变更的推荐写法

以“cleanup 增加空行删除”为例，推荐这样记录：

| 项目 | 推荐值 |
| --- | --- |
| `需求版本` | `v3.3` |
| `需求项 ID` | `REQ-CLEANUP-EMPTY-ROW` |
| `发布版本` | `3.3.4` |
| `设计说明` | cleanup 在链接预处理阶段先删除 raw 全空行 |
| `release note` | 新增 cleanup 空行删除；仅 `产品链接` 为空但其他字段有值时不会误删 |

这类改动通常不需要把需求版本提升到 `v3.4`，但如果要正式发布，就不应该继续沿用旧的 `3.3.3`。

## 8. 当前工程的简化执行规则

如果团队暂时不想一次性把流程做得很重，至少先执行下面 5 条：

1. 业务代际只看 `需求版本`，例如 `v3.3`。
2. 任何准备发布的代码变化都提升 `发布版本`。
3. `pyproject.toml` 和 `__init__.py` 中的版本必须一致。
4. 文档里的“文档版本”默认对齐当前适用的 `发布版本`。
5. 每次发布至少留下一条能说明“这个版本做了什么”的变更记录。

## 9. 后续可选增强

如果后续需要更稳的版本追踪，可以继续补这些内容：

- 新增 `CHANGELOG.md`
- 在需求文档里给关键规则补 `需求项 ID`
- 在设计文档里增加“首次实现版本 / 最近变更版本”
- 在 CI 里增加版本一致性检查
- 在发版脚本里自动校验 `pyproject.toml`、`__init__.py`、tag 是否一致
