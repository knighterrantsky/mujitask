# TikTok 链接整理与阶段一采集双入口设计

更新时间：`2026-04-01`

关联需求：`docs/business/requirement.md`

## 1. 设计目标

当前业务只开放两个独立的用户入口，不提供统一串联入口：

1. `链接整理/去重入口`
2. `TikTok 阶段一采集入口`

这样做的原因：
- 需求明确把“格式化链接并删重”定义成正式能力，而且允许用户手动调用。
- 阶段一采集只负责浏览器采集、附件上传和现有字段写回，不再混入链接清洗职责。
- 所有飞书写入必须限制在真实现有字段内，不允许新增中间字段或状态字段。

## 2. 用户入口

### 2.1 入口一：链接整理/去重

流程：
- 读取飞书行
- 提取 `产品链接`
- 规范化为 `https://www.tiktok.com/shop/pdp/{product_id}`
- 将规范 URL 回写到原字段 `产品链接`
- 基于规范 URL 判重
- 每组重复记录保留首条，删除后续重复整行

输入：
- `table_url`
- `access_token` 或 `access_token_env`
- `run_mode`
- 可选 `url_field_name`，默认 `产品链接`

输出：
- `summary`
- `items`
- `settings`

写入约束：
- keeper 行只允许回写 `产品链接`
- duplicate 行只允许删除
- 不写 `标准产品链接`
- 不写 `链接整理状态`
- 不写 `删除重复数`

### 2.2 入口二：TikTok 阶段一采集

流程：
- 读取飞书行
- 只选择已经完成链接整理的记录
- 只根据当前阶段目标字段是否为空决定是否采集
- 如果阶段一目标字段全部已有值，则跳过
- 如果阶段一目标字段存在一个或多个空缺，则抓取该条记录
- 单条记录采集成功后立即上传飞书附件
- 单条记录附件上传成功后立即写回当前记录
- 处理完当前记录后再进入下一条

输入：
- `table_url`
- `access_token` 或 `access_token_env`
- `profile_ref`
- `run_mode`
- `skip_completed_rows`
- `max_records`
- 可选 `url_field_name`，默认 `产品链接`

输出：
- `summary`
- `items`
- `settings`

前置条件：
- 该入口默认要求用户已先执行“链接整理/去重”
- 如果记录的 `产品链接` 还不是规范 URL，则直接跳过
- 如果同一规范 URL 仍存在多条记录，则直接跳过并提示先执行 cleanup

抓取触发规则：
- 阶段一只检查这些字段是否为空：
  - `SKU-ID`
  - `图片`
  - `标题`
  - `节日`
  - `卖家`
  - `前台截图`
  - `价格`
  - `记录日期`
- 如果上述字段全部有值，则标记为 `skipped_completed`
- 如果上述字段中存在一个或多个空值，则执行抓取
- 只要该条记录进入写回，本轮必须同时更新 `记录日期`，即使该字段原本已有值

执行约束：
- 阶段一入口改为流式单条执行，不保留整批采集缓存
- 不再采用“先全部 collect，再全部 upload，再全部 write_back”的模式
- 单条记录在当前轮执行中的顺序固定为：
  - 抓取
  - 上传附件
  - 直接更新飞书当前记录
- 单条记录最终失败时，必须在运行结果中返回失败记录和失败原因

写入约束：
- 仅允许写入：
  - `SKU-ID`
  - `图片`
  - `标题`
  - `节日`
  - `卖家`
  - `前台截图`
  - `价格`
  - `记录日期`
- 对于被执行更新的记录：
  - 空缺字段必须补写
  - `记录日期` 必须始终刷新为本次运行日期
- 不写 `商品主图`
- 不写 `商品页截图`
- 不写 `采集状态`
- 不写 `采集错误`
- 不写 `采集时间`

## 3. 核心模块

### 3.1 共享访问层

职责：
- 解析飞书表地址
- 读取飞书行
- 上传附件
- 更新或删除记录
- 汇总处理结果

核心输入输出：
- 输入：飞书表 URL、访问令牌、原始飞书记录
- 输出：标准化行结构、飞书可写字段、summary

### 3.2 链接整理模块

职责：
- 提取 TikTok `product_id`
- 规范化 `产品链接`
- 规划 keeper / duplicate / invalid / empty
- 构建 cleanup 写回字段

核心函数输入输出：
- 输入：`产品链接`
- 输出：规范 URL 或清洗失败原因

### 3.3 阶段一采集模块

职责：
- 浏览器打开 TikTok 商品页
- 等待页面稳定
- 抽取阶段一字段
- 生成商品主图和前台截图
- 失败时执行单条重试
- 成功后立即把结果交给当前记录写回链路，而不是先做整批缓存

核心函数输入输出：
- 输入：规范 TikTok 商品 URL、浏览器 profile
- 输出：阶段一业务数据和本地文件路径

### 3.4 阶段一落表模块

职责：
- 上传 `图片`、`前台截图`
- 生成 `记录日期`
- 构建阶段一真实写回字段
- 对当前记录立即写回飞书
- 写回失败时返回当前记录的失败原因

核心函数输入输出：
- 输入：阶段一采集结果、本地图片路径
- 输出：只含现有字段的飞书字段字典

## 4. 状态约定

### 4.1 cleanup 入口

典型状态：
- `preview`
- `updated`
- `delete_preview`
- `deleted`
- `invalid_url`
- `skipped_empty`

### 4.2 阶段一入口

典型状态：
- `preview`
- `updated`
- `failed`
- `skipped_not_cleaned`
- `skipped_duplicate_needs_cleanup`
- `skipped_completed`
- `skipped_empty`
- `invalid_url`

说明：
- 所有错误和跳过原因只保存在运行 summary 中
- 不回写飞书状态字段
- 最终失败结果必须包含 `record_id` 和 `error`

## 5. 实现约束

- 正式业务入口不再依赖通用 `field_mapping`
- 正式阶段一入口不再使用 `requests + HTML` 作为正式采集路径
- cleanup 与阶段一是两个并列入口，不互相自动调用
- 阶段一实现必须采用“读一条、抓一条、写一条”的单条流式处理方式
- 阶段一是否抓取只由阶段一目标字段是否存在空缺决定
- FastMoss 第二阶段当前只预留接口，不纳入本轮正式执行链路
