# 设计文档

更新时间：`2026-04-07`

## 1. 设计目标

本文件用于描述 v3.0 的目标设计，并结合当前代码说明现有实现基线与设计差距。  
文档目标有三点：

1. 对齐最新需求文档中的客户入口、字段边界和业务规则。
2. 真实反映当前代码里已经存在的 task、workflow、flow 和 model。
3. 明确当前实现与 v3.0 目标设计之间的差距，避免把未实现能力写成已完成能力。

## 2. 设计视角与总览

### 2.1 交付入口

v3.0 的交付入口是 OpenClaw skills，而不是单个底层 CLI task。

客户入口分为两类：

1. 定时任务入口：每天固定时间触发飞书竞品数据更新。
2. 对话入口：用户在 OpenClaw 中通过自然语言输入关键词搜索请求。

### 2.2 目标内部流程

v3.0 的业务内部流程收敛为以下三条：

| 流程名 | 角色 | 触发来源 | 主要结果 |
| --- | --- | --- | --- |
| `链接标准化去重流` | 预处理流 | 定时任务 | 标准化 `产品链接`，去重并保留首条 |
| `商品详情更新流` | 核心更新流 | 定时任务 / 关键词搜索 | 更新 TikTok 字段、FastMoss 截图、销量字段和 `记录日期` |
| `关键词候选发现流` | 候选发现流 | OpenClaw 对话 | 从 FastMoss 搜索候选商品，只为新商品落表并触发详情更新 |

### 2.3 当前代码已有任务

当前代码中已经存在的 task 如下：

| 任务名 | workflow_id | 当前角色 |
| --- | --- | --- |
| `tiktok_product_link_cleanup` | `tiktok_product_link_cleanup_v1` | 标准化 `产品链接` 并删除重复行 |
| `tiktok_feishu_batch_sync` | `tiktok_feishu_batch_sync_v1` | 从飞书批量读取记录，抓取 TikTok 字段并回写 |
| `tiktok_feishu_single_sync` | `tiktok_feishu_single_sync_v1` | 输入单个 TikTok URL，新建一条飞书记录；遇到重复 URL 或 SKU 时跳过 |
| `fastmoss_product_sales_snapshot` | `fastmoss_product_sales_snapshot_v1` | 独立抓取 FastMoss 商品详情截图与销量数据 |

这些 task 属于当前实现基线，不等于最终产品入口。

## 3. 数据与字段设计

### 3.1 字段角色与写回边界

自动维护字段固定为以下 13 个：

- `产品链接`
- `SKU-ID`
- `图片`
- `标题`
- `节日`
- `卖家`
- `前台截图`
- `价格`
- `Fastmoss截图`
- `昨日销量`
- `近7天销量`
- `近90天销量`
- `记录日期`

客户自行维护字段固定为：

- `关键词`
- `备注`
- `开售时间`
- `第一波高峰期`
- `第二波高峰期`
- `价格趋势`

设计约束：

- 自动维护字段参与待更新判断。
- 客户自行维护字段不参与待更新判断。
- `备注` 只在关键词搜索新增商品时写入来源标记。

### 3.2 待更新判断

待更新判断完全依赖自动维护字段。

当一条飞书记录在上述 13 个自动维护字段中存在 `1` 个或 `N` 个空缺时，该记录属于待更新数据；其余字段为空不影响是否待更新。

### 3.3 FastMoss 运行时数据结构

当前代码中的 `FastMossProductSalesSnapshot` 运行时结构包含：

```json
{
  "product_id": "1732268173492064949",
  "search_url": "https://www.fastmoss.com/zh/e-commerce/search?page=1&words=1732268173492064949",
  "detail_url": "https://www.fastmoss.com/zh/e-commerce/detail/1732268173492064949",
  "product_title": "Example Product",
  "login_state": "already_logged_in",
  "yesterday_sales": "0",
  "sales_7d": "24",
  "sales_28d": "940",
  "sales_90d": "1824",
  "detail_page_screenshot_local_path": "...",
  "detail_page_screenshot_file_name": "...",
  "detail_page_screenshot_mime_type": "image/png"
}
```

v3.0 设计要求：

- `detail_page_screenshot_*` 对应飞书字段 `Fastmoss截图`。
- `yesterday_sales / sales_7d / sales_90d` 需要写入飞书。
- `sales_28d` 继续保留在程序数据结构中，但不写入飞书，作为未来扩展字段保留。

### 3.4 飞书记录索引

飞书记录索引需要支持两类主键视角：

- `by_url`：按标准化后的 `产品链接` 建索引
- `by_sku`：按 `SKU-ID` 建索引

设计上：

- 定时任务更新流以当前飞书记录为输入，不依赖额外匹配。
- 关键词候选发现流必须执行两级判重：
  先按 `SKU-ID` 判重，再按标准化后的 `产品链接` 判重。
- 构建 `by_url` 索引时，飞书表中的现有 `产品链接` 也必须先走同一套标准化逻辑，再参与比较。
- 这样可以覆盖一种边界情况：客户已经手工新增了 `产品链接`，但定时任务尚未触发、`SKU-ID` 还未补齐；此时关键词搜索仍需通过标准化后的 `产品链接` 识别为已存在商品并跳过。

## 4. 目标内部流程设计

### 4.1 链接标准化去重流

职责：

- 读取飞书记录
- 标准化 `产品链接`
- 回写标准化后的 `产品链接`
- 删除重复整行

固定步骤：

1. 读取飞书表中的记录。
2. 从 `产品链接` 提取 TikTok `product_id`。
3. 将其规范化为标准 TikTok PDP URL。
4. 将规范化结果写回 `产品链接`。
5. 按规范化后的 `产品链接` 去重。
6. 遇到重复时保留首条，删除后续重复行。

当前代码映射：

- 对应 task：`tiktok_product_link_cleanup`
- 对应 workflow：`tiktok_product_link_cleanup_v1`

### 4.2 商品详情更新流

职责：

- 基于一条飞书记录补齐 TikTok 商品字段
- 基于同一商品补齐 FastMoss 截图与销量字段
- 将自动维护字段写回当前飞书行

输入最小集：

- `record_id`
- `产品链接`
- `SKU-ID`
- `profile_ref`
- FastMoss 登录信息

固定步骤：

1. 以当前飞书记录为输入。
2. 优先使用 `产品链接` 定位商品。
3. 若 `产品链接` 为空且 `SKU-ID` 不为空，则由 `SKU-ID` 拼接 TikTok 商品链接。
4. 打开 TikTok 商品页，抓取 `SKU-ID / 图片 / 标题 / 节日 / 卖家 / 前台截图 / 价格`。
5. 基于 `SKU-ID` 直达 FastMoss 商品详情页。
6. 打开 FastMoss 详情页后先截图。
7. 将截图写入 `Fastmoss截图`。
8. 再抓取 `昨日 / 7天 / 28天 / 90天` 销量。
9. 将 `昨日销量 / 近7天销量 / 近90天销量` 写入飞书。
10. 将 `记录日期` 刷新为本次成功更新时间。

输出最小集：

- `record_id`
- `status`
- `write_fields`
- `fastmoss_snapshot`
- `updated_at`

设计约束：

- `sales_28d` 只保留在运行时结果中，不写飞书。
- `Fastmoss截图` 进入飞书附件字段。

当前实现基线：

- TikTok 商品抓取与飞书写回已存在于 `tiktok_feishu_batch_sync`
- FastMoss 截图与销量抓取已存在于 `fastmoss_product_sales_snapshot`
- 当前代码尚未把两者合并成同一条商品详情更新流

### 4.3 关键词候选发现流

职责：

- 基于关键词从 FastMoss 发现候选竞品
- 对新商品写入 `SKU-ID`
- 调用商品详情更新流补齐字段

输入最小集：

- `关键词`
- `7日销量阈值`
- `profile_ref`
- FastMoss 登录信息

固定步骤：

1. 进入 FastMoss 商品搜索页。
2. 使用 `关键词` 执行搜索。
3. 固定按 `近7天销量` 排序。
4. 固定翻页穷举搜索结果。
5. 只保留 `近7天销量 > 阈值` 的候选商品。
6. 对候选商品先按 `SKU-ID` 判断是否已存在于飞书表。
7. 如果 `SKU-ID` 未命中，再按标准化后的 `产品链接` 做第二次判重。
8. 只要命中 `SKU-ID` 或命中标准化后的 `产品链接`，都视为已存在商品并直接跳过，不更新、不重写。
9. 不存在商品先写入 `SKU-ID`。
10. 对新商品调用商品详情更新流补齐自动维护字段。
11. 对新商品在 `备注` 中写入 `通过搜索关键字：{关键词}`。

输出最小集：

- `search_keyword`
- `sales_7d_threshold`
- `matched_items`
- `skipped_existing_items`
- `inserted_items`
- `processed_items`

## 5. 模块与接口设计

### 5.1 模块职责

设计文档中的模块划分采用以下结构：

1. `Feishu 表访问与索引层`
   - 负责解析 `table_url`
   - 负责读取、创建、更新、删除飞书记录
   - 负责构建 `by_url / by_sku` 索引
   - 当前代码对象：`FeishuBitableClient`、`parse_table_url`

2. `TikTok 商品抓取层`
   - 负责标准化 TikTok URL
   - 负责抓取商品字段、生成图片与页面截图
   - 当前代码对象：`fetch_tiktok_product_record`、`fetch_tiktok_product_record_via_browser`

3. `FastMoss 商品抓取层`
   - 负责登录检测
   - 负责打开商品详情页
   - 负责抓取截图与销量
   - 当前代码对象：`fetch_fastmoss_product_sales_via_browser`、`FastMossProductSalesSnapshot`

4. `飞书写回与附件上传层`
   - 负责本地文件上传为飞书附件
   - 负责将 TikTok / FastMoss 结果映射到飞书字段
   - 当前代码对象：`_prepare_writable_fields` 及各类写回逻辑

5. `OpenClaw skills 编排层`
   - 负责定时任务触发
   - 负责自然语言入口触发
   - 负责串联 cleanup、候选发现、详情更新
   - 当前代码基线为 `skills/mujitask-tiktok-feishu-sync` 下的包装脚本

### 5.2 目标内部流程接口

目标内部流程接口按行为定义，不强绑未来 Python symbol 名：

| 内部接口 | 输入最小集 | 输出最小集 |
| --- | --- | --- |
| `link_cleanup_flow` | `table_url / access_token / url_field_name` | `summary / items / deletion_results / update_results` |
| `product_detail_update_flow` | `record_id / 产品链接 / SKU-ID / profile_ref / fastmoss credentials` | `status / record_id / write_fields / fastmoss_snapshot / updated_at` |
| `keyword_candidate_discovery_flow` | `关键词 / 7日销量阈值 / profile_ref / fastmoss credentials` | `matched_items / skipped_existing_items / inserted_items / processed_items` |

## 6. OpenClaw skills 编排设计

### 6.1 定时任务 skill

v3.0 目标设计中的定时任务 skill 编排顺序为：

1. 读取飞书表配置。
2. 执行 `链接标准化去重流`。
3. 基于自动维护字段识别待更新行。
4. 对待更新行逐条执行 `商品详情更新流`。
5. 汇总本次更新结果并返回给 OpenClaw。

### 6.2 对话触发 skill

v3.0 目标设计中的对话 skill 编排顺序为：

1. 从自然语言中提取 `关键词` 与 `7日销量阈值`。
2. 执行 `关键词候选发现流`。
3. 对新商品先写入 `SKU-ID`。
4. 对新商品执行 `商品详情更新流`。
5. 汇总新增、跳过和更新结果并返回给 OpenClaw。

### 6.3 当前 skills 实现基线

当前 `skills/mujitask-tiktok-feishu-sync` 的主入口仍以 cleanup + batch 串联为主：

1. `run_feishu_tiktok_sync.*` 先运行 `run_cleanup.*`
2. 再运行 `run_batch_sync.*`
3. 最终输出组合结果给 OpenClaw

这代表当前 skills 编排基线仍围绕 TikTok cleanup 与 TikTok batch sync 组织，尚未体现 v3.0 的 FastMoss 关键词发现流与详情更新流。

### 7.3 当前代码的真实行为边界

当前代码已经实现的关键行为包括：

1. `tiktok_product_link_cleanup`
   - 能标准化 `产品链接`
   - 能按标准化 URL 去重
   - 去重时保留首条并删除后续重复行

2. `tiktok_feishu_batch_sync`
   - 能读取飞书表记录
   - 能判断一组 TikTok 字段是否缺失
   - 能通过浏览器抓取 TikTok 商品字段
   - 能上传 `图片` 和 `前台截图`
   - 能回写当前飞书行并刷新 `记录日期`

3. `tiktok_feishu_single_sync`
   - 能输入单个 TikTok URL
   - 会先按 URL 和 SKU 查重
   - 重复时跳过
   - 不重复时创建新飞书记录

4. `fastmoss_product_sales_snapshot`
   - 能访问 FastMoss 账户中心判断登录态
   - 能在需要时执行手机号密码登录
   - 当前实现仍通过搜索页查找商品详情链接
   - 能进入详情页抓取截图、昨日、7天、28天、90天销量
   - 目前只返回结果，不写飞书

## 8. 当前实现差距

### 8.1 现状 vs 目标

1. `tiktok_feishu_batch_sync`
   - 现状：只覆盖 TikTok 商品字段与 `记录日期`
   - 目标：需要并入 FastMoss 截图、`昨日销量 / 近7天销量 / 近90天销量` 写回

2. 待更新判断
   - 现状：代码中的完成字段仍是 `SKU-ID / 图片 / 标题 / 节日 / 卖家 / 前台截图 / 价格 / 记录日期`
   - 目标：待更新判断应扩大到 13 个自动维护字段，并纳入 `产品链接 / Fastmoss截图 / 昨日销量 / 近7天销量 / 近90天销量`

3. FastMoss 抓取入口
   - 现状：`fastmoss_product_sales_snapshot` 是独立 task，未并入飞书更新流程
   - 目标：FastMoss 抓取要成为 `商品详情更新流` 的固定子步骤

4. FastMoss 导航方式
   - 现状：当前 flow 仍是“搜索页 -> 详情页”
   - 目标：URL 更新流应按 `SKU-ID` 直接拼接 FastMoss detail URL

5. 关键词搜索能力
   - 现状：当前代码中还没有围绕关键词搜索、7天销量排序、翻页穷举、候选筛选、已存在跳过的完整实现链路
   - 目标：需要形成独立的 `关键词候选发现流`

6. OpenClaw skills 编排
   - 现状：主入口脚本仍是 cleanup + batch 串联
   - 目标：skills 需要升级为“定时任务更新流”和“对话关键词发现流”两类入口模型

### 8.2 当前不应误写为“已完成”的能力

以下能力属于 v3.0 目标设计，但当前代码尚未完整实现，不应在设计文档中写成“现已完成”：

- TikTok 字段与 FastMoss 字段的一体化单条更新流
- 基于 `SKU-ID` 直达 FastMoss detail URL 的更新流
- 关键词搜索候选发现与飞书写回的完整闭环
- OpenClaw 面向客户的最终 skill 入口模型
