# 设计文档

更新时间：`2026-04-10`

## 1. 设计目标

本文件用于描述 v3.3 的目标设计，并结合当前代码说明现有实现基线与设计差距。  
文档目标有三点：

1. 对齐最新需求文档中的客户入口、字段边界和业务规则。
2. 真实反映当前代码里已经存在的 task、workflow、flow 和 model。
3. 明确当前实现与 v3.3 目标设计之间的差距，避免把未实现能力写成已完成能力。

## 2. 设计视角与总览

### 2.1 交付入口

v3.3 的交付入口是 OpenClaw skills，而不是单个底层 CLI task。

客户入口分为两类：

1. 定时任务入口：每天固定时间触发飞书竞品数据更新。
2. 对话入口：用户在 OpenClaw 中通过自然语言输入关键词搜索请求。

### 2.2 目标内部流程

v3.3 的业务内部流程收敛为以下三条：

| 流程名 | 角色 | 触发来源 | 主要结果 |
| --- | --- | --- | --- |
| `链接标准化去重流` | 预处理流 | 定时任务 | 标准化 `产品链接`，去重并保留首条 |
| `商品详情更新流` | 核心更新流 | 定时任务 / 关键词搜索 | 补齐缺失的自动维护字段；若商品页明确不可访问，则写 `商品状态` 并停止后续抓取；有实际写入时刷新 `记录日期` |
| `关键词候选发现流` | 候选发现流 | OpenClaw 对话 | 从 FastMoss 搜索候选商品，只为新商品写入 `SKU-ID + 产品链接 + 备注` 并触发详情更新 |

## 3. 数据与字段设计

### 3.1 字段角色与写回边界

自动维护字段固定为以下 14 个：

- `产品链接`
- `SKU-ID`
- `图片`
- `标题`
- `节日`
- `卖家`
- `前台截图`
- `价格`
- `fastmoss价格`
- `Fastmoss截图`
- `昨日销量`
- `近7天销量`
- `近90天销量`
- `记录日期`

客户自行维护字段固定为：

- `关键词`
- `备注`
- `开售时间`
- `第一波高峰期`
- `第二波高峰期`
- `价格趋势`

系统状态字段固定为：

- `商品状态`

设计约束：

- 自动维护字段参与待更新判断。
- 客户自行维护字段和系统状态字段都不参与待更新判断。
- `价格` 固定表示 TikTok 商品页价格。
- `fastmoss价格` 固定表示 FastMoss 商品详情页价格。
- `商品状态` 固定用于标记 `已下架/区域不可售`。
- 当一条记录已经带有 `商品状态=已下架/区域不可售` 时，后续扫描和详情更新默认直接跳过。
- 如果人工修改了 `产品链接` 或 `SKU-ID`，需要同时人工清空 `商品状态`，让该行重新进入抓取流程。
- `备注` 只在关键词搜索新增商品时写入来源标记。

### 3.2 待更新判断

待更新判断完全依赖自动维护字段。

当一条飞书记录在上述 14 个自动维护字段中存在 `1` 个或 `N` 个空缺时，该记录属于待更新数据；其余字段为空不影响是否待更新。

补充约束：

- `商品状态` 自身为空或非空，都不改变 14 个自动维护字段的完整性判断。
- 但如果 `商品状态=已下架/区域不可售`，则该记录在 `pending_rows_scan` 中应直接表现为 `skipped_unavailable`，不进入后续详情更新输入集合。

### 3.3 FastMoss 运行时数据结构

当前代码中的 `FastMossProductSalesSnapshot` 运行时结构包含：

```json
{
  "product_id": "1732268173492064949",
  "search_url": "https://www.fastmoss.com/zh/e-commerce/search?page=1&words=1732268173492064949",
  "detail_url": "https://www.fastmoss.com/zh/e-commerce/detail/1732268173492064949",
  "product_title": "Example Product",
  "login_state": "skipped_login_verification",
  "fastmoss_price_amount": "29.91",
  "yesterday_sales": "0",
  "sales_7d": "24",
  "sales_28d": "940",
  "sales_90d": "1824",
  "detail_page_screenshot_local_path": "...",
  "detail_page_screenshot_file_name": "...",
  "detail_page_screenshot_mime_type": "image/png"
}
```

v3.3 设计要求：

- `fastmoss_price_amount` 对应飞书字段 `fastmoss价格`。
- `fastmoss_price_amount` 的值不带货币符号或单位，类型与 TikTok 商品页抓取到的 `price_amount` 保持一致。
- `detail_page_screenshot_*` 对应飞书字段 `Fastmoss截图`。
- `yesterday_sales / sales_7d / sales_90d` 需要写入飞书。
- `sales_28d` 继续保留在程序数据结构中，但不写入飞书，作为未来扩展字段保留。
- `login_state` 当前已存在的语义值包括 `skipped_login_verification / already_logged_in / logged_in / relogged_in_after_session_loss`。
- 当前默认抓取路径不会先访问账户中心校验登录态，而是直接打开商品详情页；仅显式传 `verify_fastmoss_login=true` 时，才先执行账户中心登录校验。

### 3.4 飞书记录索引

飞书记录索引需要支持两类主键视角：

- `by_url`：按标准化后的 `产品链接` 建索引
- `by_sku`：按 `SKU-ID` 建索引

设计上：

- 定时任务更新流以当前飞书记录为输入，不依赖额外匹配。
- 关键词候选发现流必须执行两级判重：
  先按 `SKU-ID` 判重，再按标准化后的 `产品链接` 判重。
- 构建 `by_url` 索引时，飞书表中的现有 `产品链接` 也必须先走同一套标准化逻辑，再参与比较。
- 这样可以覆盖一种边界情况：客户已经手工新增了 `产品链接`，但定时任务尚未触发、`SKU-ID` 还未补齐；此时关键词搜索仍需通过标准化后的 `产品链接` 识别为已存在商品并跳过。

## 4. 目标内部流程设计

### 4.1 链接标准化去重流

职责：

- 读取飞书记录
- 标准化 `产品链接`
- 回写标准化后的 `产品链接`
- 删除空行
- 删除重复整行

固定步骤：

1. 读取飞书表中的记录。
2. 先检查该行原始字段是否全部为空；若为空则标记为 `delete_empty` 并进入删除分支。
3. 对非空行，从 `产品链接` 提取 TikTok `product_id`。
4. 将其规范化为标准 TikTok PDP URL。
5. 将规范化结果写回 `产品链接`。
6. 按规范化后的 `产品链接` 去重。
7. 遇到重复时保留首条，删除后续重复行。

空行定义：

- 以 cleanup 阶段读取到的飞书原始字段字典为准；
- 如果该字典下所有字段都没有有效值，则该行视为空行；
- 如果只有 `产品链接` 为空，但其他字段仍有内容，则该行不属于空行，只能进入后续正常判定，不应直接删除。

当前代码映射：

- 对应 task：`tiktok_product_link_cleanup`
- 对应 workflow：`tiktok_product_link_cleanup_v1`

### 4.2 商品详情更新流

职责：

- 基于一条飞书记录补齐 TikTok 商品字段
- 基于同一商品补齐 FastMoss 价格、截图与销量字段
- 只补齐当前缺失的自动维护字段

输入最小集：

- `record_id`
- `产品链接`
- `SKU-ID`
- `profile_ref`
- FastMoss 登录信息

固定步骤：

1. 以当前飞书记录为输入。
2. 读取当前飞书行中的 `商品状态`，作为防御性保护判断。
3. 若该行 `商品状态=已下架/区域不可售`，则直接返回 `skipped_unavailable`，不进入浏览器抓取。
4. 再按 14 个自动维护字段判断该行是否仍属于待更新。
5. 若该行字段已完整，则直接返回 `skipped_completed`，不进入浏览器抓取。
6. 若该行仍待更新，则优先使用 `产品链接` 定位商品。
7. 若 `产品链接` 为空且 `SKU-ID` 不为空，则由 `SKU-ID` 拼接 TikTok 商品链接。
8. 打开 TikTok 商品页。
9. 如果页面明确表现为商品已下架、商品不存在或区域不可售，则仅写回 `商品状态=已下架/区域不可售`，并终止本条后续抓取。
10. 只有在商品页可正常访问时，才抓取 `SKU-ID / 图片 / 标题 / 节日 / 卖家 / 前台截图 / 价格`。
11. 只有在当前仍缺 `fastmoss价格 / Fastmoss截图 / 昨日销量 / 近7天销量 / 近90天销量` 其中至少一个字段时，才基于 `SKU-ID` 进入 FastMoss 商品详情页。
12. 默认直接打开 FastMoss 商品详情页；只有显式传 `verify_fastmoss_login=true` 时，才会先访问账户中心校验登录态。
13. 如果详情页识别到游客权限提示，或抓取过程中 session 丢失，则自动执行登录并重新打开详情页继续当前抓取步骤。
14. 打开 FastMoss 详情页后先截图。
15. 将截图写入 `Fastmoss截图`。
16. 再抓取 `fastmoss价格 / 昨日 / 7天 / 28天 / 90天` 销量。
17. 仅对当前为空的自动维护字段执行写回，不覆盖已有内容。
18. 只有本次确实有字段被写入时，才刷新 `记录日期`。

输出最小集：

- `record_id`
- `status`
- `write_fields`
- `fastmoss_snapshot`
- `updated_at`
- `unavailable_reason`

设计约束：

- `sales_28d` 只保留在运行时结果中，不写飞书。
- `fastmoss价格` 来自 FastMoss 商品详情页价格区域，默认取 `价格：` 标签后的当前价。
- 运行时字段 `fastmoss_price_amount` 需要去掉货币符号或单位，只保留数值字符串，例如 `$29.91` 需写成 `29.91`。
- 不写原价、划线价、运费或佣金率。
- `Fastmoss截图` 进入飞书附件字段。
- `昨日销量` 通过 FastMoss 日期控件获取，不直接向日期输入框手动输入文本。
- `昨日销量` 的日期选择策略固定为：优先选择昨天，昨天不可选则选择前天，前天也不可选则写入 `-1`。
- `产品链接` 的标准化回写属于 `链接标准化去重流`；`商品详情更新流` 只在 `产品链接` 为空时补写，不覆盖已有链接。
- 如果 TikTok 商品页明确不可访问，则本条不进入 FastMoss，也不刷新 `记录日期`。
- 如果当前缺失字段仅落在 TikTok 侧字段，而 FastMoss 相关字段已完整，则本条不再重复进入 FastMoss。
- FastMoss 详情抓取默认跳过账户中心前置登录校验，直接打开详情页。
- FastMoss 详情页检测到游客态，或抓取中途掉登录时，需要自动补登录并回到当前详情页继续执行。
- TikTok 商品抓取阶段需要兼容登录促销弹窗阻塞，并在 dismiss 前引入随机延迟以降低机械操作痕迹。
- 详情更新流不覆盖已有 `备注`，也不负责自动清空人工保留的 `商品状态`。

当前实现基线：

- 当前代码已经提供独立 task：`feishu_single_row_update`
- 该 task 已实现 TikTok 抓取、FastMoss `价格 + 截图 + 销量` 抓取、缺失字段补写与 `记录日期` 条件刷新
- 完整行会在进入浏览器前直接返回 `skipped_completed`
- 当前代码已落地 `商品状态` 的判定、写回和后续跳过逻辑
- 当前代码已落地“仅缺 TikTok 字段时跳过 FastMoss”的优化
- 当前代码已落地 TikTok 登录促销弹窗的 blocker handling 与随机延迟处理

### 4.3 关键词候选发现流

职责：

- 基于关键词从 FastMoss 发现候选竞品
- 对新商品写入 `SKU-ID + 标准化产品链接 + 备注`
- 调用商品详情更新流补齐字段

输入最小集：

- `关键词`
- `7日销量阈值`
- `profile_ref`
- FastMoss 登录信息

固定步骤：

1. 进入 FastMoss 商品搜索页。
2. 使用 `关键词` 执行搜索。
3. 固定按 `近7天销量` 排序。
4. 固定翻页穷举搜索结果。
5. 只保留 `近7天销量 > 阈值` 的候选商品。
6. 对候选商品先按 `SKU-ID` 判断是否已存在于飞书表。
7. 如果 `SKU-ID` 未命中，再按标准化后的 `产品链接` 做第二次判重。
8. 只要命中 `SKU-ID` 或命中标准化后的 `产品链接`，都视为已存在商品并直接跳过，不更新、不重写。
9. 不存在商品先写入 `SKU-ID`、按 `SKU-ID` 拼接并标准化后的 `产品链接`，以及 `备注` 来源标记。
10. 对新商品调用商品详情更新流补齐自动维护字段。
11. 如果新商品详情页明确不可访问，则保留已写入的 `SKU-ID + 产品链接 + 备注`，并补写 `商品状态=已下架/区域不可售`。
12. `商品详情更新流` 后续不再覆盖已有 `备注`。

输出最小集：

- `search_keyword`
- `sales_7d_threshold`
- `matched_items`
- `skipped_existing_items`
- `inserted_items`
- `processed_items`

## 5. 模块与接口设计

### 5.1 模块职责

设计文档中的模块划分采用以下结构：

1. `Feishu 表访问与索引层`
   - 负责解析 `table_url`
   - 负责读取、创建、更新、删除飞书记录
   - 负责构建 `by_url / by_sku` 索引
   - 当前代码对象：`FeishuBitableClient`、`parse_table_url`

2. `TikTok 商品抓取层`
   - 负责标准化 TikTok URL
   - 负责抓取商品字段、生成图片与页面截图
   - 负责判定商品页是否属于明确不可访问状态
   - 当前代码对象：`fetch_tiktok_product_record`、`fetch_tiktok_product_record_via_browser`

3. `FastMoss 商品抓取层`
   - 负责按需登录检测与会话恢复
   - 负责默认直开商品详情页
   - 负责抓取价格、截图与销量
   - 当前代码对象：`fetch_fastmoss_product_sales_via_browser`、`FastMossProductSalesSnapshot`

4. `飞书写回与附件上传层`
   - 负责本地文件上传为飞书附件
   - 负责将 TikTok / FastMoss 结果映射到飞书字段
   - 负责在商品不可访问时仅写回 `商品状态`
   - 当前代码对象：`_prepare_writable_fields` 及各类写回逻辑

5. `OpenClaw skills 编排层`
   - 负责定时任务触发
   - 负责自然语言入口触发
   - 负责串联 cleanup、候选发现、详情更新
   - 当前代码基线为 `skills/mujitask-tiktok-feishu-sync` 下的包装脚本

| `tiktok_feishu_single_sync` | `product_url / table_url / access_token_env / write_back` | `status / record_id / product_url / product_id / duplicate_reason` | 单条 URL 插入能力 |
| `fastmoss_product_sales_snapshot` | `product_id or sku_id / profile_ref / fastmoss credentials / capture_detail_screenshot` | `fastmoss_snapshot` | 独立 FastMoss 抓取能力 |

## 6. OpenClaw skills 编排设计

### 6.1 定时任务 skill

v3.3 目标设计中的定时任务 skill 编排顺序为：

1. 读取飞书表配置。
2. 执行 `链接标准化去重流`。
3. 基于自动维护字段识别待更新行。
4. 对待更新行逐条执行 `商品详情更新流`。
5. 汇总本次更新结果并返回给 OpenClaw。

补充规则：

- 已经被标记为 `已下架/区域不可售` 的行，不进入本轮详情更新。
- 如果客户人工修改了该行的 `产品链接` 或 `SKU-ID`，应先人工清空 `商品状态`，再由后续定时任务重新抓取。

### 6.2 对话触发 skill

v3.3 目标设计中的对话 skill 编排顺序为：

1. 从自然语言中提取 `关键词` 与 `7日销量阈值`。
2. 执行 `关键词候选发现流`。
3. 对新商品先写入 `SKU-ID + 产品链接 + 备注`。
4. 对新商品执行 `商品详情更新流`。
5. 汇总新增、跳过和更新结果并返回给 OpenClaw。

补充规则：

- 如果新商品的 TikTok 商品页被判定为不可访问，则保留已写入的 `SKU-ID + 产品链接 + 备注`，并补写 `商品状态=已下架/区域不可售`。

### 6.3 当前 skills 实现基线

当前 `skills/mujitask-tiktok-feishu-sync` 目录已经收敛为 step 脚本集合：

- `run_fastmoss_login_check_step.sh`
- `run_cleanup_step.sh`
- `run_pending_rows_step.sh`
- `run_single_row_update_step.sh`
- `run_keyword_candidate_step.sh`
- `run_insert_seed_row_step.sh`

这代表当前代码已经切到 step 化编排基线，不再继续维护旧的全表 batch 入口脚本。

### 7.3 当前代码的真实行为边界

当前代码已经实现的关键行为包括：

1. `tiktok_product_link_cleanup`
   - 能识别 raw 所有字段都为空的空行
   - 能删除空行，且仅 `产品链接` 为空但其他字段有值时不会误删
   - 能标准化 `产品链接`
   - 能按标准化 URL 去重
   - 去重时保留首条并删除后续重复行

2. `feishu_pending_rows_scan`
   - 能按 14 个自动维护字段识别待更新行
   - 自动维护字段完整的行会直接标记为 `skipped_completed`
   - 已标记 `商品状态=已下架/区域不可售` 的行会直接标记为 `skipped_unavailable`

3. `feishu_single_row_update`
   - 能在进入浏览器前跳过已完整行
   - 能在进入浏览器前跳过已标记为 `已下架/区域不可售` 的行
   - 能对待更新行抓取 TikTok 和 FastMoss 数据
   - 能在 TikTok 明确不可访问时仅写回 `商品状态`
   - 能在只缺 TikTok 字段时跳过 FastMoss
   - 只补当前缺失字段，不覆盖已有内容
   - 只有在本次确实有字段写入时才刷新 `记录日期`

4. `feishu_seed_row_insert`
   - 能为关键词搜索新商品写入 `SKU-ID + 产品链接 + 备注`
   - 已存在商品会直接跳过，不重复插入

5. `fastmoss_keyword_candidate_discovery`
   - 能固定按 `近7天销量` 排序并翻页穷举
   - 能按 `SKU-ID + 标准化产品链接` 做两级判重
   - 命中已存在商品时直接跳过

6. `tiktok_feishu_single_sync`
   - 能输入单个 TikTok URL
   - 会先按 URL 和 SKU 查重
   - 重复时跳过
   - 不重复时创建新飞书记录

7. `fastmoss_product_sales_snapshot`
   - 默认直开 FastMoss 商品详情页
   - 显式传 `verify_fastmoss_login=true` 时能先访问 FastMoss 账户中心判断登录态
   - 能在需要时执行手机号密码登录
   - 能识别详情页游客权限文案，并自动补登录后回到详情页继续执行
   - 能在抓取中途 session 失效时自动补登录并恢复到当前详情页
   - 能按 `SKU-ID` 直达 FastMoss 详情页
   - 能进入详情页抓取 `fastmoss价格`、截图、昨日、7天、28天、90天销量
   - 目前只返回结果，不写飞书

8. `fetch_tiktok_product_record_via_browser`
   - 能识别商品页明确不可访问状态
   - 能处理 TikTok 登录促销弹窗阻塞
   - 弹窗 dismiss 前带有随机延迟

补充说明：

- 当前代码中 `feishu_competitor_flow` 仍复用 `tiktok_feishu_sync_flow` 中的部分表访问与写回辅助函数。
- `tiktok_feishu_sync_flow` 当前只保留 `tiktok_feishu_single_sync` 这类低层单条 URL 能力，不再承担 v3 主链路的产品入口职责。

## 8. 当前实现差距

### 8.1 现状 vs 目标

1. 端到端编排
   - 现状：`feishu_pending_rows_scan / feishu_single_row_update / feishu_seed_row_insert / fastmoss_keyword_candidate_discovery` 已拆成独立步骤
   - 目标：仍需要由 OpenClaw skill 将这些步骤稳定编排成定时任务链路和关键词链路

2. OpenClaw skills 编排
   - 现状：主入口逻辑已经切到 `cleanup -> pending_rows_scan -> single_row_update` 的 step 编排
   - 目标：skills 需要升级为“定时任务更新流”和“对话关键词发现流”两类入口模型

3. 共享 helper 归位
   - 现状：`feishu_competitor_flow` 仍依赖 `tiktok_feishu_sync_flow` 中的公共 helper
   - 目标：将 v3 主链路依赖的表访问、URL/写回辅助函数迁到独立共享模块，避免继续向旧 flow 聚合新业务逻辑

### 8.2 当前不应误写为“已完成”的能力

以下能力属于 v3.3 目标设计，但当前代码尚未完整实现，不应在设计文档中写成“现已完成”：

- OpenClaw 面向客户的最终 skill 入口模型
- `feishu_competitor_flow` 与 `tiktok_feishu_sync_flow` 的 helper 解耦
