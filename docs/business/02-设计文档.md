## 2. 设计视角与总览

### 2.1 当前交付入口

当前对外入口是 OpenClaw skill，而不是底层 leaf task。

当前对外入口已经有三类正式入口：

1. 竞品表刷新入口：提交顶层任务 `refresh_current_competitor_table`
2. 关键词搜索入口：提交顶层任务 `search_keyword_competitor_products`
3. 达人池同步入口：提交或执行顶层任务 `sync_tk_influencer_pool`

三类入口的共同点：

- skill 只负责意图识别、参数提取和顶层 submit
- 首条回执同步返回 `request_id`
- 后续执行与最终通知由后台运行时推进

其中 `sync_tk_influencer_pool` 当前支持三种运行形态：

- `queue_mode=inline`：默认形态，单次 run 内扫描来源记录，并小批量 drain 达人详情 job。
- `queue_mode=dispatch_only`：只扫描来源表并派发 `product_job`。
- `queue_mode=worker`：作为队列 worker 消费 `product / author / finalizer` job。

### 2.2 当前业务内部流程

当前业务内部流程以已交付能力和后续演进目标合并观察，可收敛为以下四条：

| 流程名 | 角色 | 触发来源 | 主要结果 |
| --- | --- | --- | --- |
| `链接标准化去重流` | 预处理流 | 定时任务 | 标准化 `产品链接`，去重并保留首条 |
| `商品详情更新流` | 核心更新流 | 定时任务 / 关键词搜索 | 补齐缺失的自动维护字段；若商品页明确不可访问，则写 `商品状态` 并停止后续抓取；有实际写入时刷新 `记录日期` |
| `关键词候选发现流` | 候选发现流 | OpenClaw 对话 | 从 FastMoss 搜索候选商品，只为新商品写入 `SKU-ID + 产品链接 + 备注` 并触发详情更新 |
| `达人池同步流` | 达人沉淀流 | 定时任务 | 从 `TK竞品收集` 扫描待处理商品，筛选达人并 upsert 到 `TK达人池` |

### 2.3 当前代码任务分层

当前代码中的 task 可以分成两层：

顶层业务任务：

| 任务名 | workflow_id | 当前角色 |
| --- | --- | --- |
| `refresh_current_competitor_table` | `refresh_current_competitor_table_v1` | 顶层竞品表刷新任务 |
| `search_keyword_competitor_products` | `search_keyword_competitor_products_v1` | 顶层关键词搜索任务 |
| `sync_tk_influencer_pool` | `sync_tk_influencer_pool_v1` | 顶层达人池同步任务，支持 inline / dispatch_only / worker 形态 |

叶子或辅助任务：

| 任务名 | workflow_id | 当前角色 |
| --- | --- | --- |
| `fastmoss_login_check` | `fastmoss_login_check_v1` | 单次校验 FastMoss 登录态 |
| `feishu_pending_rows_scan` | `feishu_pending_rows_scan_v1` | 按 14 个自动维护字段识别待更新行 |
| `feishu_single_row_update` | `feishu_single_row_update_v1` | 以单条飞书记录为单位执行 TikTok + FastMoss 补全 |
| `feishu_seed_row_insert` | `feishu_seed_row_insert_v1` | 为关键词搜索发现的新商品写入种子行 |
| `fastmoss_keyword_candidate_discovery` | `fastmoss_keyword_candidate_discovery_v1` | 在 FastMoss 搜索候选商品并执行两级判重 |
| `tiktok_product_link_cleanup` | `tiktok_product_link_cleanup_v1` | 标准化 `产品链接` 并删除空行、重复行 |
| `tiktok_feishu_single_sync` | `tiktok_feishu_single_sync_v1` | 输入单个 TikTok URL，新建一条飞书记录；遇到重复 URL 或 SKU 时跳过 |
| `fastmoss_product_sales_snapshot` | `fastmoss_product_sales_snapshot_v1` | 独立抓取 FastMoss 商品详情截图与销量数据 |

说明：

- 当前正式产品入口已经不是这些 leaf task 的直接拼接。
- 竞品表刷新、关键词搜索和达人池同步三条主链路都已经通过顶层任务进入统一执行平台。
- 达人池的 `product / author / finalizer` worker 不是独立对外 task，而是同一个 `sync_tk_influencer_pool` 在不同 `queue_mode / worker_kinds` 参数下的内部运行形态。
- `sync_tk_influencer_pool_single_row_v1` 和 `sync_tk_influencer_pool_author_detail` 属于内部队列 item / job 语义，不作为客户直接调用入口。

## 3. 数据与字段设计

### 3.1 字段角色与写回边界

自动维护字段固定为以下 14 个：

- `产品链接`
- `SKU-ID`
- `图片`
- `标题`
- `节日`
- `卖家`
- `前台截图`
- `价格`
- `fastmoss价格`
- `Fastmoss截图`
- `昨日销量`
- `近7天销量`
- `近90天销量`
- `记录日期`

客户自行维护字段固定为：

- `关键词`
- `备注`
- `开售时间`
- `第一波高峰期`
- `第二波高峰期`
- `价格趋势`

系统状态字段固定为：

- `商品状态`

设计约束：

- 自动维护字段参与待更新判断。
- 客户自行维护字段和系统状态字段都不参与待更新判断。
- `价格` 固定表示 TikTok 商品页价格。
- `fastmoss价格` 固定表示 FastMoss 商品详情页价格。
- `商品状态` 固定用于标记 `已下架/区域不可售`。
- 当一条记录已经带有 `商品状态=已下架/区域不可售` 时，后续扫描和详情更新默认直接跳过。
- 如果人工修改了 `产品链接` 或 `SKU-ID`，需要同时人工清空 `商品状态`，让该行重新进入抓取流程。
- `备注` 只在关键词搜索新增商品时写入来源标记。

### 3.2 待更新判断

待更新判断完全依赖自动维护字段。

当一条飞书记录在上述 14 个自动维护字段中存在 `1` 个或 `N` 个空缺时，该记录属于待更新数据；其余字段为空不影响是否待更新。

补充约束：

- `商品状态` 自身为空或非空，都不改变 14 个自动维护字段的完整性判断。
- 但如果 `商品状态=已下架/区域不可售`，则该记录在 `pending_rows_scan` 中应直接表现为 `skipped_unavailable`，不进入后续详情更新输入集合。

### 3.3 FastMoss 运行时数据结构

当前代码中的 `FastMossProductSalesSnapshot` 运行时结构包含：

```json
{
  "product_id": "1732268173492064949",
  "search_url": "https://www.fastmoss.com/zh/e-commerce/search?page=1&words=1732268173492064949",
  "detail_url": "https://www.fastmoss.com/zh/e-commerce/detail/1732268173492064949",
  "product_title": "Example Product",
  "login_state": "skipped_login_verification",
  "fastmoss_price_amount": "29.91",
  "yesterday_sales": "0",
  "sales_7d": "24",
  "sales_28d": "940",
  "sales_90d": "1824",
  "detail_page_screenshot_local_path": "...",
  "detail_page_screenshot_file_name": "...",
  "detail_page_screenshot_mime_type": "image/png"
}
```

v3.3 设计要求：

- `fastmoss_price_amount` 对应飞书字段 `fastmoss价格`。
- `fastmoss_price_amount` 的值不带货币符号或单位，类型与 TikTok 商品页抓取到的 `price_amount` 保持一致。
- `detail_page_screenshot_*` 对应飞书字段 `Fastmoss截图`。
- `yesterday_sales / sales_7d / sales_90d` 需要写入飞书。
- `sales_28d` 继续保留在程序数据结构中，但不写入飞书，作为未来扩展字段保留。
- `login_state` 当前已存在的语义值包括 `skipped_login_verification / already_logged_in / logged_in / relogged_in_after_session_loss`。
- 当前默认抓取路径不会先访问账户中心校验登录态，而是直接打开商品详情页；仅显式传 `verify_fastmoss_login=true` 时，才先执行账户中心登录校验。

### 3.4 飞书记录索引

飞书记录索引需要支持两类主键视角：

- `by_url`：按标准化后的 `产品链接` 建索引
- `by_sku`：按 `SKU-ID` 建索引

设计上：

- 定时任务更新流以当前飞书记录为输入，不依赖额外匹配。
- 关键词候选发现流必须执行两级判重：
  先按 `SKU-ID` 判重，再按标准化后的 `产品链接` 判重。
- 构建 `by_url` 索引时，飞书表中的现有 `产品链接` 也必须先走同一套标准化逻辑，再参与比较。
- 这样可以覆盖一种边界情况：客户已经手工新增了 `产品链接`，但定时任务尚未触发、`SKU-ID` 还未补齐；此时关键词搜索仍需通过标准化后的 `产品链接` 识别为已存在商品并跳过。

### 3.5 达人池字段边界与写回规则

`TK达人池` 继续保持一人一行，主键固定为 `达人ID`。

自动维护字段固定为：

- `带货商品图`
- `关联节日`
- `关联商品销量`
- `达人头像`
- `粉丝数`
- `28天视频数`
- `带货视频 GMV`
- `带货直播 GMV`
- `合作店铺`
- `合作商品数`
- `达人联系方式`
- `记录时间`

非自动维护字段固定为：

- `检查达人名称是否重复`
- `跟我们合作过的节日`
- `出爆款视频，但是没有合作过`
- `毕业季建联`
- `达人地址`
- `达人电话`

字段类型约束：

- 附件字段：`带货商品图`、`达人头像`
- 多选字段：`关联节日`、`合作店铺`
- 文本字段：`关联商品销量`、`粉丝数`、`28天视频数`、`带货视频 GMV`、`带货直播 GMV`、`合作商品数`、`达人联系方式`
- 日期时间字段：`记录时间`

设计约束：

- `检查达人名称是否重复` 是公式字段，不参与写回。
- 来源 `TK竞品收集.达人查找状态` 只服务于同步状态管理，不属于达人池画像字段。
- `TK达人池` 的多商品聚合幂等不依赖飞书隐藏字段，而依赖运行时内部状态。

## 4. 目标内部流程设计

### 4.1 链接标准化去重流

职责：

- 读取飞书记录
- 标准化 `产品链接`
- 回写标准化后的 `产品链接`
- 删除空行
- 删除重复整行

固定步骤：

1. 读取飞书表中的记录。
2. 先检查该行原始字段是否全部为空；若为空则标记为 `delete_empty` 并进入删除分支。
3. 对非空行，从 `产品链接` 提取 TikTok `product_id`。
4. 将其规范化为标准 TikTok PDP URL。
5. 将规范化结果写回 `产品链接`。
6. 按规范化后的 `产品链接` 去重。
7. 遇到重复时保留首条，删除后续重复行。

空行定义：

- 以 cleanup 阶段读取到的飞书原始字段字典为准；
- 如果该字典下所有字段都没有有效值，则该行视为空行；
- 如果只有 `产品链接` 为空，但其他字段仍有内容，则该行不属于空行，只能进入后续正常判定，不应直接删除。

当前代码映射：

- 对应 task：`tiktok_product_link_cleanup`
- 对应 workflow：`tiktok_product_link_cleanup_v1`

### 4.2 商品详情更新流

职责：

- 基于一条飞书记录补齐 TikTok 商品字段
- 基于同一商品补齐 FastMoss 价格、截图与销量字段
- 只补齐当前缺失的自动维护字段

输入最小集：

- `record_id`
- `产品链接`
- `SKU-ID`
- `profile_ref`
- FastMoss 登录信息

固定步骤：

1. 以当前飞书记录为输入。
2. 读取当前飞书行中的 `商品状态`，作为防御性保护判断。
3. 若该行 `商品状态=已下架/区域不可售`，则直接返回 `skipped_unavailable`，不进入浏览器抓取。
4. 再按 14 个自动维护字段判断该行是否仍属于待更新。
5. 若该行字段已完整，则直接返回 `skipped_completed`，不进入浏览器抓取。
6. 若该行仍待更新，则优先使用 `产品链接` 定位商品。
7. 若 `产品链接` 为空且 `SKU-ID` 不为空，则由 `SKU-ID` 拼接 TikTok 商品链接。
8. 打开 TikTok 商品页。
9. 如果页面明确表现为商品已下架、商品不存在或区域不可售，则仅写回 `商品状态=已下架/区域不可售`，并终止本条后续抓取。
10. 只有在商品页可正常访问时，才抓取 `SKU-ID / 图片 / 标题 / 节日 / 卖家 / 前台截图 / 价格`。
11. 只有在当前仍缺 `fastmoss价格 / Fastmoss截图 / 昨日销量 / 近7天销量 / 近90天销量` 其中至少一个字段时，才基于 `SKU-ID` 进入 FastMoss 商品详情页。
12. 默认直接打开 FastMoss 商品详情页；只有显式传 `verify_fastmoss_login=true` 时，才会先访问账户中心校验登录态。
13. 如果详情页识别到游客权限提示，或抓取过程中 session 丢失，则自动执行登录并重新打开详情页继续当前抓取步骤。
14. 打开 FastMoss 详情页后先截图。
15. 将截图写入 `Fastmoss截图`。
16. 再抓取 `fastmoss价格 / 昨日 / 7天 / 28天 / 90天` 销量。
17. 仅对当前为空的自动维护字段执行写回，不覆盖已有内容。
18. 只有本次确实有字段被写入时，才刷新 `记录日期`。

输出最小集：

- `record_id`
- `status`
- `write_fields`
- `fastmoss_snapshot`
- `updated_at`
- `unavailable_reason`

设计约束：

- `sales_28d` 只保留在运行时结果中，不写飞书。
- `fastmoss价格` 来自 FastMoss 商品详情页价格区域，默认取 `价格：` 标签后的当前价。
- 运行时字段 `fastmoss_price_amount` 需要去掉货币符号或单位，只保留数值字符串，例如 `$29.91` 需写成 `29.91`。
- 不写原价、划线价、运费或佣金率。
- `Fastmoss截图` 进入飞书附件字段。
- `昨日销量` 通过 FastMoss 日期控件获取，不直接向日期输入框手动输入文本。
- `昨日销量` 的日期选择策略固定为：优先选择昨天，昨天不可选则选择前天，前天也不可选则写入 `-1`。
- `产品链接` 的标准化回写属于 `链接标准化去重流`；`商品详情更新流` 只在 `产品链接` 为空时补写，不覆盖已有链接。
- 如果 TikTok 商品页明确不可访问，则本条不进入 FastMoss，也不刷新 `记录日期`。
- 如果当前缺失字段仅落在 TikTok 侧字段，而 FastMoss 相关字段已完整，则本条不再重复进入 FastMoss。
- FastMoss 详情抓取默认跳过账户中心前置登录校验，直接打开详情页。
- FastMoss 详情页检测到游客态，或抓取中途掉登录时，需要自动补登录并回到当前详情页继续执行。
- TikTok 商品抓取阶段需要兼容登录促销弹窗阻塞，并在 dismiss 前引入随机延迟以降低机械操作痕迹。
- 详情更新流不覆盖已有 `备注`，也不负责自动清空人工保留的 `商品状态`。

当前实现基线：

- 当前代码已经提供独立 task：`feishu_single_row_update`
- 该 task 已实现 TikTok 抓取、FastMoss `价格 + 截图 + 销量` 抓取、缺失字段补写与 `记录日期` 条件刷新
- 完整行会在进入浏览器前直接返回 `skipped_completed`
- 当前代码已落地 `商品状态` 的判定、写回和后续跳过逻辑
- 当前代码已落地“仅缺 TikTok 字段时跳过 FastMoss”的优化
- 当前代码已落地 TikTok 登录促销弹窗的 blocker handling 与随机延迟处理

### 4.3 关键词候选发现流

职责：

- 基于关键词从 FastMoss 发现候选竞品
- 对新商品写入 `SKU-ID + 标准化产品链接 + 备注`
- 调用商品详情更新流补齐字段

输入最小集：

- `关键词`
- `7日销量阈值`
- `profile_ref`
- FastMoss 登录信息

固定步骤：

1. 进入 FastMoss 商品搜索页。
2. 使用 `关键词` 执行搜索。
3. 固定按 `近7天销量` 排序。
4. 固定翻页穷举搜索结果。
5. 只保留 `近7天销量 > 阈值` 的候选商品。
6. 对候选商品先按 `SKU-ID` 判断是否已存在于飞书表。
7. 如果 `SKU-ID` 未命中，再按标准化后的 `产品链接` 做第二次判重。
8. 只要命中 `SKU-ID` 或命中标准化后的 `产品链接`，都视为已存在商品并直接跳过，不更新、不重写。
9. 不存在商品先写入 `SKU-ID`、按 `SKU-ID` 拼接并标准化后的 `产品链接`，以及 `备注` 来源标记。
10. 对新商品调用商品详情更新流补齐自动维护字段。
11. 如果新商品详情页明确不可访问，则保留已写入的 `SKU-ID + 产品链接 + 备注`，并补写 `商品状态=已下架/区域不可售`。
12. `商品详情更新流` 后续不再覆盖已有 `备注`。

输出最小集：

- `search_keyword`
- `sales_7d_threshold`
- `matched_items`
- `skipped_existing_items`
- `inserted_items`
- `processed_items`

### 4.4 达人池同步流

职责：

- 从 `TK竞品收集` 扫描待处理竞品
- 基于 FastMoss 商品页达人列表和达人详情补齐 `TK达人池`
- 通过来源表状态字段管理同步进度

输入最小集：

- `table_url`
- `target_table_url`
- `access_token_env`
- `fastmoss_phone_env`
- `fastmoss_password_env`

固定步骤：

1. 扫描 `TK竞品收集` 中 `达人查找状态` 为 `空值 / 待查找 / 失败重试 / 处理中` 的记录。
2. 跳过 `商品状态=已下架/区域不可售` 的记录。
3. 初始化 FastMoss HTTP Session、来源表 target、目标表 target、目标表字段 schema 和达人 `entity_snapshot` 索引。
4. 每条来源记录开始处理前先写 `达人查找状态=处理中`。
5. 用 `SKU-ID` 作为 FastMoss `product_id`，调 `GET /api/goods/v3/author`，固定 `pagesize=10`。
6. 在商品达人列表阶段先按 `sold_count > 50` 和列表可见粉丝数 `> 5000` 过滤。
7. 对达标达人创建或恢复 `author_job`，唯一语义为 `source_record_id + product_id + influencer_id`。
8. 达人详情阶段按 job 拉取 `baseInfo / authorIndex / cargoSummary / shopList`；只有显式传入 `include_contact=true` 时才额外拉取 `authorContact`。
9. 写回前先检查 `entity_snapshot.source_product_ids`，如果同一 `达人ID + product_id` 已成功写过，则直接跳过详情写回，避免重复追加商品图和重复累加销量。
10. 以 `达人ID` 为主键 upsert `TK达人池`，并写入或更新达人 `entity_snapshot`。
11. `inline` 默认会在单次来源记录处理中 drain 一批 author job，默认上限为 `max_author_detail_jobs_per_source_row=50`。
12. `dispatch_only` 只派发 `product_job`；`worker` 形态再按 `product / author / finalizer` 三类 worker 消费队列。
13. finalizer 根据同一来源商品下 author job 状态聚合结果，并回写来源记录 `达人查找状态=已完成 / 失败重试`。
14. 任一达人详情接口命中 `MAG_AUTH_3002` 时，当前 run hard-stop，已成功写入的达人 checkpoint 保留，不回滚。

字段来源：

- `带货商品图` 来自来源竞品记录 `图片`
- `关联商品销量` 来自商品页该达人对应的 `sold_count`
- `关联节日` 来自来源竞品记录 `节日`
- `达人头像` 来自 `baseInfo.avatar`
- `粉丝数`、`28天视频数` 来自 `authorIndex`
- `带货视频 GMV`、`带货直播 GMV` 来自 `cargoSummary`
- `合作店铺` 来自 `shopList`
- `达人联系方式` 来自 `authorContact`，但当前为可选补充字段；只有 `include_contact=true` 时才尝试获取，拿不到稳定正文时不阻塞其他字段写回
- `记录时间` 来自本次成功写回日期时间

合并规则：

- `TK达人池` 继续保持一人一行。
- 同一商品重复跑时，不重复追加商品图，不重复累加 `关联商品销量` 和 `合作商品数`。
- 同一达人命中新商品时，才追加商品图、累加销量、合并节日、合并店铺并将 `合作商品数 + 1`。

幂等方案：

- 不依赖飞书隐藏字段。
- 依赖内部 `entity_snapshot` 保存 `source_product_ids / source_product_sales_by_id / holiday_names / cooperation_shop_names`。

当前队列拆分与目标演进：

为避免单条 `SKU-ID` 命中几十到上百个达人时导致顶层任务长时间运行，达人池同步已经引入数据库队列驱动的二级任务模型。当前代码支持 `dispatch_only` 派发、`worker` drain 和默认 `inline` 小批量 drain；长期目标是把 worker 进程独立部署成持续运行或定时运行的后台消费者。

1. 表扫描与商品任务派发
   - 读取 `TK竞品收集`，按 `达人查找状态`、`商品状态` 和后续刷新策略筛选需要处理的竞品行。
   - 每条候选竞品写入 `influencer_pool_product_job`，建议唯一键为 `source_record_id + product_id`。
   - 该阶段只创建或恢复商品级任务，不直接拉取达人详情，也不直接写 `TK达人池`。

2. 商品达人列表 worker
   - worker 扫描并 claim `influencer_pool_product_job` 中 `pending / failed_retry` 的任务。
   - 对单个 `product_id` 调 `GET /api/goods/v3/author` 拉取商品达人列表。
   - 在列表阶段完成进入下一环节的筛选，筛选条件为 `sold_count > 50` 且 `follower_count > 5000` 同时满足。
   - 达标达人只写入内部 `influencer_pool_author_job`，建议唯一键为 `source_record_id + product_id + influencer_id`。
   - 该阶段只写数据库，不写飞书；即使单个商品命中大量达人，也主要消耗少量分页请求和数据库 upsert。

3. 达人详情 worker
   - worker 扫描并 claim `influencer_pool_author_job` 中 `pending / failed_retry` 的任务。
   - 每个 author job 只负责一个 `influencer_id + product_id`，拉取 `baseInfo / authorIndex / cargoSummary / shopList`，并在 `include_contact=true` 时额外尝试 `authorContact`，合并列表阶段已有的商品销量、商品图、节日等上下文。
   - 成功后 upsert `TK达人池`，写入或更新 `entity_snapshot`，并将 author job 标记为 `succeeded`。
   - 失败时只标记当前 author job 为 `failed_retry` 并记录达人级 trace，不应阻塞同商品下其它达人任务。

4. 商品任务完成判定与回写
   - finalizer 根据同一 `source_record_id + product_id` 下所有 author job 状态聚合商品级结果。
   - 全部 author job 为 `succeeded / skipped` 时，回写来源记录 `达人查找状态=已完成`。
   - 存在可重试失败时，回写来源记录 `达人查找状态=失败重试`，等待下轮只补失败 author job。
   - 汇总新增、更新、跳过、失败数量后写入 notification outbox，由现有 outbox dispatcher 发送飞书通知。

队列插入边界：

- `dispatch_only` 形态只插入或恢复 `product_job`。
- 商品列表 worker 插入或恢复 `author_job`。
- 达人详情 worker 只消费 `author_job`、写飞书和 snapshot，不反向创建商品任务。
- finalizer 只负责商品级状态聚合、来源表回写和通知 outbox。
- 所有 worker 都通过数据库任务状态和租约字段实现 claim，避免同一任务被并发重复处理。

推荐状态模型：

- `product_job.status`：`pending / discovering / detail_pending / detail_running / completed / failed_retry / hard_stopped`。
- `author_job.status`：`pending / running / succeeded / skipped / failed_retry / hard_failed`。
- `author_job.trace` 至少包含 `influencer_id / product_id / stage / field / url / code / error_type / error`。

达人级重试 checkpoint：

- `author_job` 的唯一键保证同一 `influencer_id + product_id` 不重复派发。
- `entity_snapshot.source_product_ids` 继续作为写入成功 checkpoint；拉详情前如果已包含当前 `product_id`，则 author job 可直接标记为 `skipped`，不再重复拉详情、不再重复写飞书。
- 同一商品下已经成功写入的达人不因后续达人失败而回滚；下一轮只重试 `failed_retry` 的 author job。

硬停止规则：

- 任一达人详情接口返回 `FastMossHTTPError.response_code == "MAG_AUTH_3002"` 时，当前 run 立即 hard-stop。
- 当前 author job 记录 `MAG_AUTH_3002 / fastmoss_permission / stage / url / error`，当前 product job 标记为 `failed_retry` 或 `hard_stopped`。
- hard-stop 前已成功写入的达人保留 checkpoint，不回滚。
- 本次 run 不继续处理后续 product job，并写入飞书汇总通知，包含已新增、已更新、已跳过、失败条目和失败原因。

当前实现说明：

- 当前端到端实现已经具备达人级 checkpoint、失败 trace、`MAG_AUTH_3002` hard-stop、`author_job` 小批量 drain、`dispatch_only` 和 `worker` 模式。
- 目标形态应进一步把 `product_job_worker`、`author_job_worker` 和 finalizer 配置为独立后台进程或稳定定时任务，让表任务主要负责发现与派发，不同步等待达人详情任务全部完成。

设计约束：

- FastMoss 数据请求主路径走纯 HTTP；浏览器恢复相关参数和状态钩子仅作为登录态或风控兜底预留，不作为当前主路径。
- FastMoss 请求全部走纯 HTTP Session，并默认严格生成 `_time / cnonce / fm-sign`。
- `fd_tk` 只存在于单次任务运行时 Session 中，不进入任何持久化接口。
- 当前 inline 模式仍由顶层任务逐条处理来源记录，但达人详情已经落到 author job，并受 `max_author_detail_jobs_per_source_row` 控制。
- 当前 dispatch / worker 模式已经使用 `influencer_pool_product_job`、`influencer_pool_author_job` 和租约字段做数据库队列。
- 接口失败按“任务级隔离 + 重试 + 错误记录”处理，避免整批阻塞。
- 来源表读取失败、目标表字段元数据读取失败或 FastMoss 初始化登录失败属于全局错误，应直接结束整次任务。
- 单行商品缺少 `SKU-ID`、单行达人详情失败或单行飞书写回失败属于行级错误，应只将当前来源记录标记为 `失败重试` 并继续后续记录。

## 6. OpenClaw skills 编排设计

### 6.1 定时任务 skill

v3.3 目标设计中的定时任务 skill 编排顺序为：

1. 读取飞书表配置。
2. 执行 `链接标准化去重流`。
3. 基于自动维护字段识别待更新行。
4. 对待更新行逐条执行 `商品详情更新流`。
5. 汇总本次更新结果并返回给 OpenClaw。

补充规则：

- 已经被标记为 `已下架/区域不可售` 的行，不进入本轮详情更新。
- 如果客户人工修改了该行的 `产品链接` 或 `SKU-ID`，应先人工清空 `商品状态`，再由后续定时任务重新抓取。

### 6.2 对话触发 skill

v3.3 目标设计中的对话 skill 编排顺序为：

1. 从自然语言中提取 `关键词` 与 `7日销量阈值`。
2. 执行 `关键词候选发现流`。
3. 对新商品先写入 `SKU-ID + 产品链接 + 备注`。
4. 对新商品执行 `商品详情更新流`。
5. 汇总新增、跳过和更新结果并返回给 OpenClaw。

补充规则：

- 如果新商品的 TikTok 商品页被判定为不可访问，则保留已写入的 `SKU-ID + 产品链接 + 备注`，并补写 `商品状态=已下架/区域不可售`。

### 6.3 达人池定时同步 skill

当前达人池定时同步 skill 已接入三类命令：

1. `influencer-pool-sync-submit`
   - 读取来源 `TK竞品收集` 和目标 `TK达人池` 的表配置。
   - 返回同步提交回执；当前仍是轻量 submit 占位能力，用于对齐 OpenClaw 异步入口形态。

2. `influencer-pool-sync`
   - 直接执行顶层任务 `sync_tk_influencer_pool`。
   - 默认 `queue_mode=inline`，会扫描来源记录、创建 author job，并在单次 run 内小批量处理达人详情。
   - 可通过 `--queue-mode dispatch_only` 改为只派发 product job。

3. `influencer-pool-worker`
   - 直接执行同一个顶层任务，但固定传入 `queue_mode=worker`。
   - 默认 `worker_kinds=product,author,finalizer`，可按需要只跑其中一种 worker。
   - 用于消费 `influencer_pool_product_job / influencer_pool_author_job` 并执行 finalizer。

补充规则：

- `include_contact` 默认关闭；只有显式传 `--include-contact` 才尝试获取 `authorContact`。
- `max_author_detail_jobs_per_source_row` 控制 inline 模式单条来源记录最多处理多少个达人详情 job，避免单条记录拖垮整次 run。
- worker 模式支持 `worker_max_iterations`、`worker_stop_when_idle` 等参数，当前适合定时触发或小批量 drain；长期目标是配置成稳定后台消费者。

### 7.3 当前代码的真实行为边界

当前代码已经实现的关键行为包括：

1. `tiktok_product_link_cleanup`
   - 能识别 raw 所有字段都为空的空行
   - 能删除空行，且仅 `产品链接` 为空但其他字段有值时不会误删
   - 能标准化 `产品链接`
   - 能按标准化 URL 去重
   - 去重时保留首条并删除后续重复行

2. `feishu_pending_rows_scan`
   - 能按 14 个自动维护字段识别待更新行
   - 自动维护字段完整的行会直接标记为 `skipped_completed`
   - 已标记 `商品状态=已下架/区域不可售` 的行会直接标记为 `skipped_unavailable`

3. `feishu_single_row_update`
   - 能在进入浏览器前跳过已完整行
   - 能在进入浏览器前跳过已标记为 `已下架/区域不可售` 的行
   - 能对待更新行抓取 TikTok 和 FastMoss 数据
   - 能在 TikTok 明确不可访问时仅写回 `商品状态`
   - 能在只缺 TikTok 字段时跳过 FastMoss
   - 只补当前缺失字段，不覆盖已有内容
   - 只有在本次确实有字段写入时才刷新 `记录日期`

4. `feishu_seed_row_insert`
   - 能为关键词搜索新商品写入 `SKU-ID + 产品链接 + 备注`
   - 已存在商品会直接跳过，不重复插入

5. `fastmoss_keyword_candidate_discovery`
   - 能固定按 `近7天销量` 排序并翻页穷举
   - 能按 `SKU-ID + 标准化产品链接` 做两级判重
   - 命中已存在商品时直接跳过

6. `tiktok_feishu_single_sync`
   - 能输入单个 TikTok URL
   - 会先按 URL 和 SKU 查重
   - 重复时跳过
   - 不重复时创建新飞书记录

7. `fastmoss_product_sales_snapshot`
   - 默认直开 FastMoss 商品详情页
   - 显式传 `verify_fastmoss_login=true` 时能先访问 FastMoss 账户中心判断登录态
   - 能在需要时执行手机号密码登录
   - 能识别详情页游客权限文案，并自动补登录后回到详情页继续执行
   - 能在抓取中途 session 失效时自动补登录并恢复到当前详情页
   - 能按 `SKU-ID` 直达 FastMoss 详情页
   - 能进入详情页抓取 `fastmoss价格`、截图、昨日、7天、28天、90天销量
   - 目前只返回结果，不写飞书

8. `fetch_tiktok_product_record_via_browser`
   - 能识别商品页明确不可访问状态
   - 能处理 TikTok 登录促销弹窗阻塞
   - 弹窗 dismiss 前带有随机延迟

9. `sync_tk_influencer_pool`
   - 能扫描 `达人查找状态` 为空、`待查找`、`失败重试`、`处理中` 的来源记录
   - 能以 `SKU-ID` 调 FastMoss 商品达人列表接口，并按 `sold_count > 50`、列表粉丝数 `> 5000` 过滤
   - 能创建或恢复 `influencer_pool_author_job`
   - 能在 inline 模式下小批量 drain author job，默认每条来源记录最多处理 50 个达人详情 job
   - 能在 `dispatch_only` 模式下只派发 product job
   - 能在 `worker` 模式下消费 `product / author / finalizer` job
   - 能按 `达人ID` upsert `TK达人池`，并写入 `entity_snapshot` 作为达人级 checkpoint
   - 能通过 finalizer 聚合 author job 状态并回写来源表 `达人查找状态`
   - 能在命中 `MAG_AUTH_3002` 时 hard-stop 当前 run，并保留已成功写入的 checkpoint
   - 能按配置写 notification outbox，供后续通知链路消费

补充说明：

- 当前代码中 `feishu_competitor_flow` 仍复用 `tiktok_feishu_sync_flow` 中的部分表访问与写回辅助函数。
- `tiktok_feishu_sync_flow` 当前只保留 `tiktok_feishu_single_sync` 这类低层单条 URL 能力，不再承担 v3 主链路的产品入口职责。
- 当前代码已具备达人池纯 HTTP 同步、`dispatch_only` 派发和 `worker` drain 的核心链路；但后台 worker 的稳定部署、自动调度和可观测性仍属于后续演进，不应误写为“已完成”。

## 8. 当前实现差距

### 8.1 现状 vs 目标

1. 端到端编排
   - 现状：`feishu_pending_rows_scan / feishu_single_row_update / feishu_seed_row_insert / fastmoss_keyword_candidate_discovery` 已拆成独立步骤
   - 目标：仍需要由 OpenClaw skill 将这些步骤稳定编排成定时任务链路和关键词链路

2. OpenClaw skills 编排
   - 现状：主入口逻辑已经切到 `cleanup -> pending_rows_scan -> single_row_update` 的 step 编排
   - 目标：skills 需要升级为“定时任务更新流”和“对话关键词发现流”两类入口模型

3. 共享 helper 归位
   - 现状：`feishu_competitor_flow` 仍依赖 `tiktok_feishu_sync_flow` 中的公共 helper
   - 目标：将 v3 主链路依赖的表访问、URL/写回辅助函数迁到独立共享模块，避免继续向旧 flow 聚合新业务逻辑

4. 达人池纯 HTTP 会话层
   - 现状：当前达人池链路已走纯 HTTP Session，并默认生成 `_time / cnonce / fm-sign`
   - 目标：继续完善登录态恢复、权限错误识别、请求重试与可观测性

5. 达人池扫描与同步 flow
   - 现状：当前已有 `TK竞品收集 -> TK达人池` 的扫描、状态流转、达人 upsert、失败重试、达人级 checkpoint 和初步 author job 小批量处理
   - 目标：从“单次 run 内 drain 小批量 author job”演进为独立 `product_job_worker / author_job_worker / finalizer`

6. 字段类型适配
   - 现状：当前已具备达人池主要字段映射、附件上传、记录时间写回和头像 404 非阻塞处理
   - 目标：继续增强字段 schema 变更兼容、附件失败降级和字段级 trace

7. 达人级幂等状态
   - 现状：当前已使用 `entity_snapshot.source_product_ids` 作为 `influencer_id + product_id` 成功 checkpoint
   - 目标：继续完善商品-达人维度的可刷新策略、历史版本和任务级审计

8. 达人池数据库队列 worker
   - 现状：当前已有 product job / author job 持久化、`dispatch_only` 派发、`worker` 小批量 drain 和 finalizer
   - 目标：配置长期运行或定时触发的后台 worker，并补齐运维监控、异常告警和部署说明

### 8.2 当前不应误写为“已完成”的能力

以下能力属于当前目标设计，但当前代码尚未完整实现，不应在设计文档中写成“现已完成”：

- OpenClaw 面向客户的最终 skill 入口模型
- `feishu_competitor_flow` 与 `tiktok_feishu_sync_flow` 的 helper 解耦
- 达人池后台 worker 的长期运行部署、自动调度和可观测性
- 达人池商品-达人维度的定期刷新策略
- 达人池任务级审计、失败 trace 查询页和通知消费闭环
