## 1. 设计原则

### 1.1 分层原则

这套库表按 4 层理解最清楚：

1. 执行控制层
   - `task_request`
   - `task_execution`
   - `resource_lease`
   - `notification_outbox`
2. 运行产物层
   - `artifact_object`
3. 业务实体层
   - `entity_registry`
   - `external_binding`
   - `entity_snapshot`
4. 外部对象层
   - 飞书记录
   - TikTok 商品链接
   - FastMoss 页面与结果

### 1.2 三类数据不要混

这三类数据必须分开：

- 执行状态
  - 谁在跑
  - 跑到哪一步
  - 是否失败
- 业务事实
  - 商品标题
  - 价格
  - 销量
  - 卖家
- 原始运行产物
  - 截图
  - HTML
  - stdout
  - 原始 JSON

对应存放位置：

- 执行状态放 `task_request / task_execution / resource_lease / notification_outbox`
- 业务事实放 `entity_snapshot.facts_jsonb`
- 原始运行产物放 `artifact_object + MinIO`

### 1.3 `entity` 的基本定义

这里最容易误解，所以先讲清楚：

- `entity` 不是“一次抓取结果”
- `entity` 是稳定业务对象
- 一次抓取的结果，通常是该对象的一次 `snapshot`

用当前业务举例：

- 同一个 TikTok 商品，在今天、明天、后天各抓一次
- 这 3 次抓取不会生成 3 个不同的 `entity`
- 它们应该对应：
  - 1 个 `product entity`
  - 3 条 `entity_snapshot`

## 4. 字段类型约定

| 类型场景 | 推荐类型 |
| --- | --- |
| 主键与业务 ID | `text` |
| 结构化参数 | `jsonb` |
| 时间字段 | `timestamptz` |
| 业务归档日期 | `date` |
| 计数与重试次数 | `integer` |
| 队列序号与文件大小 | `bigint` |

说明：

- 所有业务 ID 由应用层生成，不依赖数据库自增。
- `jsonb` 只承接结构化但不稳定的事实或参数，不承担大文件存储。

## 6. `entity` 设计详解

### 6.1 为什么不能直接把飞书记录当实体

因为飞书记录只是某个业务表里的一个外部承载对象，不是稳定业务对象本身。

举例：

- 同一个商品，可能在飞书里出现多条记录
- 同一个商品，可能来自不同关键词搜索批次
- 同一个商品，可能先被人工录入，再被自动同步命中

如果直接把飞书记录当实体，就会出现：

- 同一商品被误认为多个对象
- 后续无法做实体级历史追踪
- 商品事实和飞书记录耦合过深

所以这里的关系应该是：

- 商品本身是 `entity`
- 飞书记录是 `external_binding`
- 某次采集后的业务事实是 `entity_snapshot`

### 6.2 一个 `entity` 到底表示什么

当前最重要的是 `product entity`。

它表示：

- 一个稳定存在的商品对象
- 它的身份由 `entity_type + canonical_key` 决定

建议主键语义：

- `entity_type = product`
- `canonical_key = tiktok_product:<product_id>`

如果当前阶段拿不到稳定 `product_id`，可以临时退化为：

- `canonical_key = normalized_pdp_url:<normalized_url>`

但这是低优先级 fallback，不是长期首选。

### 6.3 `canonical_key` 的选择优先级

建议优先级：

1. 平台稳定商品 ID
2. 经过标准化后的唯一商品 URL
3. 平台 + 店铺 + 商品标题 + 其他辅助特征拼接键

原因很直接：

- 价格会变
- 标题会改
- 飞书记录会删改
- 但平台商品 ID 通常最稳定

### 6.4 `entity_registry` 只放“身份”，不放大量业务事实

`entity_registry` 的职责很克制，只放：

- 这个实体是谁
- 它是否存在
- 最近一次看到它是什么时候
- 最新快照是哪一条

不建议把这些高波动字段直接塞进 `entity_registry`：

- 标题
- 当前价格
- 当前销量
- 当前卖家名
- 当前节日标签

这些都应该进 `entity_snapshot.facts_jsonb`。

### 6.5 `entity_snapshot` 放什么

`entity_snapshot` 存的是某次抓取后的结构化业务事实。

适合放进去的内容：

- 标题
- 卖家
- TikTok 价格
- FastMoss 价格
- 昨日销量
- 7 日销量
- 90 日销量
- 节日
- 商品状态

不适合放进去的内容：

- 截图二进制
- 原始 HTML 文件
- stdout 全量日志
- 浏览器 session 数据

这些都应进入：

- `artifact_object`
- MinIO

### 6.6 什么时候生成 `entity_snapshot`

当前建议按最简单也最稳的口径：

- 只要一次业务采集拿到了可用事实，就生成一条新 `entity_snapshot`

这意味着：

- 成功执行 `feishu_single_row_update`，通常生成一条 snapshot
- `cleanup` 这种纯数据修正步骤，不生成 snapshot
- `feishu_seed_row_insert` 如果只是插入外部记录但没有拿到完整业务事实，也不一定生成 snapshot

后续如果需要做优化，可以再加：

- 相同事实快照去重
- 仅发生事实变化时才生成新 snapshot

但那是优化，不是第一版前提。

### 6.7 一个实体可以绑定多个外部对象

这正是 `external_binding` 存在的意义。

一个 `product entity` 可以同时绑定：

- 一个飞书记录
- 另一个飞书表里的镜像记录
- 一个 TikTok PDP URL
- 一个 FastMoss 页面来源键

但一个外部对象只能绑定到一个实体。

所以约束是：

- `external_binding(target_type, target_space, target_id)` 唯一

这能避免：

- 同一条飞书记录被重复绑定到两个商品实体

### 6.8 一个完整例子

假设用户通过飞书触发“补全当前竞品表”，其中一条记录：

- `record_id = rec_abcd`
- TikTok 商品 ID = `1731098351299629802`

推荐的数据落点是：

1. `task_request`
   - 代表这次“补全当前竞品表”的顶层任务
2. `task_execution`
   - 代表对 `rec_abcd` 这条记录执行 `feishu_single_row_update`
3. `entity_registry`
   - 新增或复用一条：
   - `entity_type = product`
   - `canonical_key = tiktok_product:1731098351299629802`
4. `external_binding`
   - 绑定这条实体与飞书记录：
   - `target_type = feishu_record`
   - `target_space = app_xxx.tbl_xxx`
   - `target_id = rec_abcd`
5. `entity_snapshot`
   - 保存这次抓取的标题、价格、销量、卖家等事实
6. `artifact_object`
   - 记录这次运行的截图、stdout、HTML、原始 JSON 等索引

### 6.9 一句话总结

对当前系统来说，最准确的理解是：

- `entity` 是稳定业务对象
- `entity_snapshot` 是一次抓取后的结构化事实
- `external_binding` 是它和飞书记录、外部 URL、外部 ID 的关系
- `artifact_object` 是那次运行留下来的原始证据和产物
