# 系统升级收敛方案

更新时间：`2026-04-12`

## 1. 收敛结论

本次系统升级不再走“先重写业务代码、先做一层大而全抽象平台”的路线。

当前仓库更适合被理解为一个“CLI 驱动的业务自动化 scaffold”：

- 已经有稳定的 CLI 入口
- 已经有 OpenClaw 对接脚本
- 已经有成熟的 runtime 文件协议
- 已经有 `__OPENCLAW_RESULT__` 输出约定

因此，本次最可实施的升级方案是：

**保留现有 CLI / workflow / runtime 协议，在外层新增常驻执行器，并引入 Postgres 作为控制面、MinIO 作为对象存储。**

## 2. 当前系统的真实形态

结合仓库现状和现有运行说明，当前系统有 4 个稳定事实。

### 2.1 业务入口是 CLI / skill，不是服务化 API

当前最稳定的边界是：

- `automation_business_scaffold.cli`
- OpenClaw skill step 脚本
- `__OPENCLAW_RESULT__` 返回协议

这意味着系统主轴已经不是“HTTP 服务驱动业务”，而是“CLI 和步骤脚本驱动业务”。

### 2.2 运行协议已经成熟

当前 runtime 已经形成一套稳定约定：

- `run.json`
- `steps.json`
- `signals.json`
- `stdout.log`
- `artifacts/<run_id>/<step_id>/state.json`
- 最终 `__OPENCLAW_RESULT__ <json>`

这套协议本身已经是一套小型执行框架，不应该轻易推翻。

### 2.3 当前真正稳定的业务只有两条

现阶段对外稳定提供的是两类入口：

- 更新当前飞书竞品表
- 按关键词搜索 FastMoss 商品并写入当前飞书表

也就是说，当前稳定主轴仍然是“商品链路”，而不是五张表全面平台化。

### 2.4 飞书五表更像业务视图，不是底层真源

当前五表本质上仍可归纳为两条链路：

- 商品链：`TK选品收集 -> TK竞品收集`
- 达人链：`TK合作爆款视频 -> TK达人池 -> TK达人建联表`

并且表间关系主要仍是软关联。当前最稳定的键是：

- 商品：`商品ID / SKU-ID / PDP URL 提取 ID`
- 达人：`达人ID`
- 视频：`视频码`

所以这一阶段最需要保住的不是飞书宽表结构，而是：

- CLI 入口
- workflow 步骤脚本
- runtime 记录协议

## 3. 本次升级的目标形态

目标系统收敛为“3 层 + 1 个对象存储”。

### 3.1 OpenClaw 入口层

职责只有两个：

- 接收用户 skill 输入
- 插入待执行任务

这一层不直接占用浏览器，不直接等待完整业务跑完。

### 3.2 Resident Executor 常驻执行层

这是本次新增的核心层，负责：

- 轮询待执行任务
- 给浏览器 `profile` 加锁
- 串行执行 TikTok 相关任务
- 调用现有 CLI / workflow
- 收集结果
- 持久化状态
- 触发通知

### 3.3 数据与状态层

该层拆成两部分：

- `Postgres`：管理任务、资源锁、运行状态、快照索引、通知 outbox
- `MinIO`：管理运行文件、截图、HTML、stdout、`state.json`、下载物

### 3.4 对象存储 MinIO

MinIO 不是用来替代数据库，而是承接当前 runtime 文件体系的长期存储。

它最适合存放：

- `run.json`
- `steps.json`
- `signals.json`
- `stdout.log`
- `state.json`
- 截图
- HTML 快照
- 下载文件
- 原始抓取 JSON

## 4. 目标调用链

Phase 1 先包住旧代码，不先重写业务逻辑。

```text
OpenClaw skill
-> submit task
-> executor daemon
-> subprocess 调现有 cli/workflow
-> 现有代码继续产出 runtime 文件与 __OPENCLAW_RESULT__
-> executor 解析结果
-> 上传 MinIO
-> 更新 Postgres
-> 发通知
```

这个路径的意义是：

- 不推翻 TikTok / FastMoss / 飞书业务逻辑
- 保留当前 runtime 目录结构
- 保留当前 OpenClaw 输出协议
- 先解决单 profile 资源竞争这个最现实的问题

## 5. 为什么这次不先做大而全的平台表

这次不建议优先数据库化的内容有：

- skill 到 workflow 的映射
- 不同业务到飞书表的映射
- 字段映射规则
- 通知模板
- channel 路由

这些在第一阶段更适合留在：

- Python 配置
- YAML 配置
- 现有文档和脚本约定

原因很简单：当前业务仍在快速变化，过早数据库化只会增加维护成本。

这一阶段真正必须入库的只有三类：

- 执行控制
- 事实快照
- 通知补偿

## 6. 推荐技术栈

推荐保持轻量：

- `Python 3.11+`
- 保留现有 CLI 入口
- `PostgreSQL`
- `SQLAlchemy + Alembic`
- `MinIO`
- `boto3` 或 `minio` SDK
- `systemd` / `supervisor` / `docker compose`

当前不建议引入：

- `Celery`
- `Kafka`
- `Airflow`
- 大拆微服务

当前最大约束不是高并发，而是单浏览器 profile 的串行占用。这个问题用常驻 executor 加 Postgres 行锁就足够解决。

## 7. 第一阶段最小数据库表

第一阶段只建议建设 8 张表。

### 7.1 `task_request`

表示一次来自 OpenClaw 的请求。

核心字段：

- `request_id`
- `project_code`
- `skill_code`
- `payload_json`
- `idempotency_key`
- `status`
- `requested_by`
- `created_at`

作用：

- OpenClaw 只写这里
- 入口层不直接碰浏览器

### 7.2 `task_execution`

表示一次真正执行的任务。

核心字段：

- `execution_id`
- `request_id`
- `workflow_code`
- `status`
- `queue_seq`
- `resource_pool_code`
- `worker_id`
- `run_id`
- `summary_json`
- `result_json`
- `error_text`
- `started_at`
- `finished_at`
- `heartbeat_at`

作用：

- 这里才是实际执行状态真源

### 7.3 `resource_lease`

表示谁占用了哪个浏览器资源。

核心字段：

- `resource_code`
- `execution_id`
- `lease_until`
- `heartbeat_at`
- `status`

当前如果只有一个 profile，可以直接约定：

- `resource_code = browser.tiktok.main`

### 7.4 `artifact_object`

记录 MinIO 对象索引。

核心字段：

- `artifact_id`
- `run_id`
- `step_id`
- `kind`
- `bucket`
- `object_key`
- `etag`
- `size`
- `content_type`
- `created_at`

`kind` 建议至少支持：

- `run_json`
- `steps_json`
- `signals_json`
- `stdout_log`
- `state_json`
- `screenshot`
- `download`
- `html_snapshot`
- `raw_capture`

### 7.5 `entity_registry`

统一实体，不按飞书表建宽表。

核心字段：

- `entity_id`
- `entity_type`
- `canonical_key`
- `status`
- `first_seen_at`
- `last_seen_at`

当前建议先支持：

- `product`

后续再扩展：

- `creator`
- `video`

### 7.6 `external_binding`

内部实体与外部系统记录的绑定关系。

核心字段：

- `binding_id`
- `entity_id`
- `target_type`
- `table_id`
- `record_id`
- `source_key`
- `updated_at`

当前主要承接：

- Feishu table record

### 7.7 `entity_snapshot`

统一快照表，用 `JSONB` 存事实，不做业务宽表。

核心字段：

- `snapshot_id`
- `entity_id`
- `snapshot_date`
- `facts_jsonb`
- `baseline_snapshot_id`
- `diff_jsonb`
- `run_id`
- `created_at`

例如 `product` 的 `facts_jsonb` 可以是：

```json
{
  "title": "xxx",
  "seller": "xxx",
  "price_tiktok": "12.99",
  "price_fastmoss": "11.99",
  "sales_1d": "20",
  "sales_7d": "180",
  "sales_90d": "1200",
  "status": "active",
  "festival": "毕业季"
}
```

### 7.8 `notification_outbox`

所有通知先写入 outbox，再异步发送。

核心字段：

- `outbox_id`
- `channel_code`
- `event_type`
- `ref_type`
- `ref_id`
- `payload_json`
- `status`
- `retry_count`
- `next_retry_at`
- `sent_at`

## 8. 为什么 MinIO 适合这套系统

本项目非常适合 MinIO，但 MinIO 只适合存“文件产物”，不适合单独承担“运行状态”。

适合进 MinIO 的内容：

- `run.json`
- `steps.json`
- `signals.json`
- `stdout.log`
- `artifacts/.../state.json`
- 截图
- 原始 HTML
- 下载图片
- 原始抓取 JSON

必须放 Postgres 的内容：

- 当前任务状态
- 队列顺序
- 资源锁
- 心跳
- 最新摘要
- 通知待发送状态

因此最合理的划分是：

- `Postgres` 存活状态
- `MinIO` 存不可变产物

## 9. MinIO 的落地方式

第一阶段不建议一步改写所有 step 的文件写入逻辑。

### 9.1 第一阶段

现有代码继续写本地 runtime：

```text
runtime/cli_runs/
runtime/artifacts/
runtime/downloads/
```

执行完成后，由 executor 统一把本次 `run_id` 对应的 runtime 文件上传到 MinIO。

### 9.2 第二阶段

再引入 `ArtifactStore` 抽象：

- `LocalArtifactStore`
- `MinioArtifactStore`

逐步把截图、`state.json`、`stdout` 从“本地直写”迁移到“抽象接口写入”。

## 10. 建议的对象前缀

建议至少划分 3 类对象前缀：

```text
mujitask-runtime/
  runs/<run_id>/run.json
  runs/<run_id>/steps.json
  runs/<run_id>/signals.json
  runs/<run_id>/stdout.log.gz
  runs/<run_id>/artifacts/<step_id>/state.json

mujitask-capture/
  raw/<source>/<yyyy>/<mm>/<dd>/<capture_id>.json
  html/<source>/<yyyy>/<mm>/<dd>/<capture_id>.html
  screenshot/<source>/<yyyy>/<mm>/<dd>/<capture_id>.png

mujitask-attachments/
  feishu/<entity_type>/<entity_id>/<filename>
```

并建议增加两条统一规则：

- 对象 metadata 中带上 `run_id / step_id / entity_id / content_type / sha256`
- 开启 versioning 和 lifecycle

## 11. `__OPENCLAW_RESULT__` 的兼容升级方向

当前返回字段主要是本地路径：

- `run_file`
- `steps_file`
- `signals_file`
- `stdout_file`

这在单机上没问题，但如果未来 executor 和 OpenClaw 不在同一机器，本地绝对路径就失去价值。

因此建议做兼容升级：

### 11.1 先保留旧字段

- `run_file`
- `steps_file`
- `signals_file`
- `stdout_file`

### 11.2 新增对象存储字段

- `artifact_uri_prefix`
- `run_object_key`
- `steps_object_key`
- `signals_object_key`
- `stdout_object_key`

这样既不破坏当前 OpenClaw 逻辑，也为后续跨机部署预留了通道。

## 12. 当前业务应该怎样接入新系统

第一阶段不要把五张表全部纳入调度中心，只先接当前最稳定的两条商品业务：

- 定时更新当前飞书竞品表
- 关键词搜索商品并写入当前飞书表

原因是这两条链路：

- 现有实现最成熟
- 主键最清晰
- 结果最容易验证

达人链和视频链可以复用同一套 executor / DB / MinIO，但不建议作为第一阶段先做。

## 13. 代码、数据库、对象存储的边界

### 13.1 继续放代码或 YAML 的

- skill -> workflow 映射
- workflow -> 飞书表映射
- 飞书字段映射
- alert 规则
- channel 路由

### 13.2 必须放数据库的

- 任务请求
- 执行状态
- 资源租约
- 外部记录绑定
- 快照
- outbox

### 13.3 必须放 MinIO 的

- runtime 文件
- artifact
- screenshot
- html
- raw capture
- download

## 14. 推荐的代码改造顺序

### Phase 1：先把“同步直跑”改成“异步排队”

建议新增：

- `submit_task.py`
- `executor_daemon.py`
- `queue_repo.py`
- `lease_repo.py`

这一阶段的 executor 仍通过 `subprocess` 调用现有 CLI。

### Phase 2：接 MinIO

建议新增：

- `artifact_store.py`
- `minio_store.py`
- `artifact_sync.py`

### Phase 3：加快照与变化

建议新增：

- `entity_service.py`
- `snapshot_service.py`
- `diff_service.py`
- `binding_service.py`

第一阶段只先做 `product`。

### Phase 4：加通知

建议新增：

- `alert_service.py`
- `outbox_dispatcher.py`
- `openclaw_channel_adapter.py`

### Phase 5：扩达人 / 视频业务

复用同一套 executor / DB / MinIO，不另起一套平台。

## 15. 最终建议

一句话概括这版方案：

**保留当前的 CLI + workflow + runtime 文件协议；新增常驻 executor 解决单 profile 竞争；用 Postgres 存控制状态和快照索引；用 MinIO 存运行产物和原始文件；第一阶段只先接商品链，不先把五表全部平台化。**

这版方案的关键不是“重新定义业务”，而是先把旧系统包住、排队、持久化、可追踪，再逐步演进到更稳定的执行平台。
