# 系统升级收敛方案

更新时间：`2026-04-13`

## 1. 收敛结论

本次系统升级不再走“先重写业务代码、先做一层大而全抽象平台”的路线。

当前仓库更适合被理解为一个“CLI 驱动的业务自动化 scaffold”：

- 已经有稳定的 CLI 入口
- 已经有 OpenClaw 对接脚本
- 已经有成熟的 runtime 文件协议
- 已经有 `__OPENCLAW_RESULT__` 输出约定

因此，本次最可实施的升级方案是：

**保留现有 CLI / workflow / runtime 协议，在外层新增常驻执行器，并引入 Postgres 作为控制面、MinIO 作为对象存储。**

## 2. 当前系统的真实形态

结合仓库现状和现有运行说明，当前系统有 4 个稳定事实。

### 2.1 业务入口是 CLI / skill，不是服务化 API

当前最稳定的边界是：

- `automation_business_scaffold.cli`
- OpenClaw skill step 脚本
- `__OPENCLAW_RESULT__` 返回协议

这意味着系统主轴已经不是“HTTP 服务驱动业务”，而是“CLI 和步骤脚本驱动业务”。

### 2.2 运行协议已经成熟

当前 runtime 已经形成一套稳定约定：

- `run.json`
- `steps.json`
- `signals.json`
- `stdout.log`
- `artifacts/<run_id>/<step_id>/state.json`
- 最终 `__OPENCLAW_RESULT__ <json>`

这套协议本身已经是一套小型执行框架，不应该轻易推翻。

### 2.3 当前真正稳定的业务只有两条

现阶段对外稳定提供的是两类入口：

- 更新当前飞书竞品表
- 按关键词搜索 FastMoss 商品并写入当前飞书表

也就是说，当前稳定主轴仍然是“商品链路”，而不是五张表全面平台化。

### 2.4 飞书五表更像业务视图，不是底层真源

当前五表本质上仍可归纳为两条链路：

- 商品链：`TK选品收集 -> TK竞品收集`
- 达人链：`TK合作爆款视频 -> TK达人池 -> TK达人建联表`

并且表间关系主要仍是软关联。当前最稳定的键是：

- 商品：`商品ID / SKU-ID / PDP URL 提取 ID`
- 达人：`达人ID`
- 视频：`视频码`

所以这一阶段最需要保住的不是飞书宽表结构，而是：

- CLI 入口
- workflow 步骤脚本
- runtime 记录协议

## 3. 本次升级的目标形态

目标系统收敛为“3 层 + 1 个对象存储”。

### 3.1 OpenClaw 入口层

职责只有两个：

- 接收用户 skill 输入
- 插入待执行任务

这一层不直接占用浏览器，不直接等待完整业务跑完。

### 3.2 Resident Executor 常驻执行层

这是本次新增的核心层，负责：

- 轮询待执行任务
- 给浏览器 `profile` 加锁
- 串行执行 TikTok 相关任务
- 调用现有 CLI / workflow
- 收集结果
- 持久化状态
- 触发通知

### 3.3 数据与状态层

该层拆成两部分：

- `Postgres`：管理任务、资源锁、运行状态、快照索引、通知 outbox
- `MinIO`：管理运行文件、截图、HTML、stdout、`state.json`、下载物

### 3.4 对象存储 MinIO

MinIO 不是用来替代数据库，而是承接当前 runtime 文件体系的长期存储。

它最适合存放：

- `run.json`
- `steps.json`
- `signals.json`
- `stdout.log`
- `state.json`
- 截图
- HTML 快照
- 下载文件
- 原始抓取 JSON

### 3.5 运行时进程结构

当前推荐的运行时结构是 3 个常驻进程 + 1 个入口层：

1. OpenClaw 入口层
2. `executor_daemon`
3. `browser_runloop`
4. `outbox_dispatcher`

它们在运行时的职责如下。

#### 3.5.1 OpenClaw 入口层

职责：

- 接收飞书或调度输入
- 生成顶层 `task_request`
- 返回“任务已受理”

它不等待整条长流程跑完，也不直接持有浏览器资源。

#### 3.5.2 `executor_daemon`

它是顶层任务编排器，不是浏览器消费 worker。

职责：

- 读取待执行 `task_request`
- 执行确定性的非浏览器步骤
- 生成需要浏览器的叶子 `task_execution`
- 在叶子任务全部终态后继续后续步骤
- 生成顶层任务汇总
- 写 `notification_outbox`

它不应该长时间阻塞在一个浏览器操作里等待整张表跑完。

#### 3.5.3 `browser_runloop`

它是浏览器任务消费 worker。

职责：

- 从 `task_execution` 中领取需要浏览器资源的叶子任务
- 按 `resource_code` 串行执行
- 获取与续租 `resource_lease`
- 调用现有 CLI / workflow
- 回写叶子任务状态

它不负责：

- 理解顶层业务语义
- 生成顶层任务汇总
- 直接给飞书发送最终通知

#### 3.5.4 `outbox_dispatcher`

它是通知发送 worker。

职责：

- 扫描 `notification_outbox`
- 发送最终业务汇总通知
- 处理发送失败重试
- 更新发送状态

它不负责：

- 执行业务任务
- 决定汇总内容

#### 3.5.5 推荐部署形态

这里说的 3 个运行组件，推荐的生产形态是：

- 同一个代码仓库
- 同一套数据库
- 3 个独立常驻进程
- 可以部署在同一台机器上，也可以后续拆到多台机器

也就是说，推荐形态不是：

- 3 个独立项目
- 3 套各自维护状态的系统

而是：

- 一个系统里的 3 个独立 worker 角色

当前推荐的入口形态可以是：

- `python -m ... executor_daemon`
- `python -m ... browser_runloop`
- `python -m ... outbox_dispatcher`

或：

- 3 个 `systemd` service
- 3 个 `supervisor` program
- 3 个 `docker compose` service

开发环境为了调试方便，可以临时提供一个“本地一键启动”脚本去同时拉起这 3 个进程。

但在架构语义上，它们仍然是：

- 3 个独立运行单元
- 不共享进程内状态
- 只通过数据库状态和可选唤醒事件协作

### 3.6 运行时协作关系

运行时推荐按下面的方式协作：

```text
OpenClaw
-> 写 task_request
-> executor_daemon 执行确定性步骤
-> executor_daemon 写浏览器 task_execution
-> browser_runloop 串行消费浏览器 task_execution
-> browser_runloop 回写 task_execution 状态
-> executor_daemon 检测当前 task_request 下所有叶子任务终态
-> executor_daemon 生成汇总并写 notification_outbox
-> outbox_dispatcher 发回飞书
```

这 3 个常驻进程的关系是：

- `executor_daemon` 负责顶层任务推进
- `browser_runloop` 负责浏览器叶子任务消费
- `outbox_dispatcher` 负责通知发送

数据库是三者之间唯一真源，不靠进程内共享内存维持状态。

#### 3.6.1 `executor_daemon` 不阻塞等待单个任务

`executor_daemon` 不应该在内存里“等待某个 `task_request` 的全部子任务完成”。

推荐方式是：

- 它只领取当前处于“可推进状态”的顶层任务
- 当顶层任务推进到“等待浏览器叶子任务完成”时，立刻提交状态并释放当前执行权
- 然后继续去处理其他可推进的顶层任务

也就是说，`executor_daemon` 不是“一个任务一个线程地陪跑到底”，而是一个状态推进器。

建议顶层 `task_request.status` 至少包含：

- `pending`
- `running`
- `waiting_children`
- `ready_for_summary`
- `success`
- `failed`

其中：

- `pending` / `running` 表示还有确定性步骤可直接推进
- `waiting_children` 表示当前只差浏览器叶子任务收敛
- `ready_for_summary` 表示可以重新由 `executor_daemon` 领取并执行汇总

这样设计后，`executor_daemon` 可以并行推进很多顶层任务，而不会因为某一个长批次任务卡住。

#### 3.6.2 `browser_runloop` 的完成触发方式

`browser_runloop` 每完成一个叶子 `task_execution` 时，本来就必须回写数据库状态。

因此推荐在同一次任务完成事务中，同时完成下面两件事：

1. 更新当前叶子任务状态
2. 检查当前 `task_request` 是否已经没有活跃叶子任务

如果已经全部终态，则把父级 `task_request` 从：

- `waiting_children`

推进为：

- `ready_for_summary`

这一步不是“browser worker 直接调用 executor”，而是“browser worker 在完成写库时顺便推进父任务状态”。

这样可以避免：

- `executor_daemon` 长时间盯着单个任务轮询
- 为了等某一个任务完成而不断全表扫描

#### 3.6.3 `outbox_dispatcher` 采用“触发优先，定时补偿”

`notification_outbox` 的发送推荐采用：

- 触发优先
- 定时补偿兜底

也就是说：

- 当 `executor_daemon` 写入新的 `notification_outbox` 时，可以顺便发出本地唤醒信号或数据库事件
- `outbox_dispatcher` 收到信号后优先立即发送
- 同时保留周期性扫描，防止唤醒事件丢失

因此 `outbox_dispatcher` 不是纯定时器，也不是纯进程内回调，而是：

- 数据库状态作为真源
- 触发式加速发送
- 定时扫描作为补偿机制

## 4. 目标调用链

Phase 1 先包住旧代码，不先重写业务逻辑。

```text
OpenClaw skill
-> submit task
-> executor daemon
-> subprocess 调现有 cli/workflow
-> 现有代码继续产出 runtime 文件与 __OPENCLAW_RESULT__
-> executor 解析结果
-> 上传 MinIO
-> 更新 Postgres
-> 发通知
```

这个路径的意义是：

- 不推翻 TikTok / FastMoss / 飞书业务逻辑
- 保留当前 runtime 目录结构
- 保留当前 OpenClaw 输出协议
- 先解决单 profile 资源竞争这个最现实的问题

## 4.1 入口与执行边界冻结

为了避免后续实现继续偏移，这里明确冻结 4 条边界。

### 4.1.1 顶层任务名词

本文不额外引入 `batch_request` 新名词。

在当前升级方案里：

- `task_request` 就是顶层业务任务
- 它代表一次来自 OpenClaw 的真实业务请求
- 它不是单条浏览器操作，也不是单个 shell 脚本调用

例如下面这些都属于一个 `task_request`：

- 更新当前飞书竞品表
- 按关键词搜索 FastMoss 商品并写入当前飞书表

### 4.1.2 OpenClaw skill 的职责

OpenClaw skill 只负责：

- 识别用户意图
- 提取本次任务参数
- 生成并提交一个顶层 `task_request`
- 在需要时查询状态或展示最终结果

OpenClaw skill 不负责：

- 直接占用浏览器
- 主导完整业务拆分
- 自己在 skill 中跑批量 `for` 循环
- 直接生成最终业务汇总通知

### 4.1.3 Executor 的职责

executor 负责：

- 读取待执行 `task_request`
- 根据 `payload_json.task_code` 决定执行计划
- 把顶层业务任务拆成一组 `task_execution`
- 只在需要浏览器 profile 的步骤上申请 `resource_lease`
- 用独立 runloop 循环消费浏览器队列
- 调用现有 CLI / workflow
- 汇总执行结果
- 写运行状态、产物索引、通知 outbox

executor 不负责：

- 理解自然语言
- 决定用户到底想做哪类业务

### 4.1.4 哪些步骤需要进入资源队列

当前设计明确要求：

- 真正进入串行队列的是“需要浏览器 profile 的执行步骤”
- 不需要浏览器的步骤不占用 `resource_lease`

因此：

- `cleanup` 这类纯数据步骤，不进入浏览器资源队列
- `pending_rows_scan` 这类纯飞书表读取步骤，不进入浏览器资源队列
- `single_row_update` 这类 TikTok / FastMoss 浏览器步骤，进入浏览器资源队列
- `keyword_candidate_discovery` 这类 FastMoss 浏览器步骤，进入浏览器资源队列

### 4.1.5 长批次任务的处理规则

当前方案明确不把“补全整张表”实现成一个必须从头跑到尾的大型浏览器长任务。

正确方式是：

- `task_request` 表示一次顶层业务请求
- executor 先执行确定性的非浏览器步骤
- 遇到大量重复性的浏览器步骤时，拆成多条 `task_execution`
- 由独立浏览器 runloop 按 `resource_code` 串行消费这些 `task_execution`
- 单条 `task_execution` 失败、超时、崩溃，不影响同一 `task_request` 下其他叶子任务继续执行
- 当同一 `task_request` 下所有 `task_execution` 都进入终态后，再生成汇总和通知

这里的终态固定包括：

- `success`
- `failed`
- `skipped`

因此系统保证的是：

- 流程能收敛
- 资源不会被长时间卡死
- 失败不会阻断后续数据继续采集

而不是：

- 单个超长浏览器 run 永不出错

### 4.1.6 进程间不使用直接回调作为真源

当浏览器叶子任务全部跑完后，概念上是“通知 executor 继续后续流程”。

但在实现上，不建议采用：

- 进程内等待 future
- browser worker 直接回调某个 executor 进程
- 依赖内存对象唤醒后续步骤

推荐实现方式是：

- `browser_runloop` 只更新数据库中的 `task_execution` 状态
- `browser_runloop` 在叶子任务完成时，可顺便把父级 `task_request` 推进到 `ready_for_summary`
- `executor_daemon` 只领取 `pending / running / ready_for_summary` 这类“当前可推进”的顶层任务
- 一旦满足条件，再继续汇总和通知

这样做的原因：

- executor 进程重启不会丢失等待状态
- browser worker 重启不会丢失结果
- executor 不会因为等待某个长批次任务而阻塞其他任务推进
- 不会出现“消息已经发了，但数据库状态没改”或“数据库状态已改，但进程没收到回调”的双写问题

后续如果需要优化延迟，可以增加：

- Postgres `LISTEN / NOTIFY`
- 本地 wakeup channel

但这些都只作为优化手段，不能替代数据库状态判断。

## 4.2 OpenClaw skill 输入输出契约

### 4.2.1 skill 输入

skill 的输入固定由 3 部分组成：

- 用户输入
  - 手工消息或定时触发语义
- 通道上下文
  - `channel_code`
  - `account_id`
  - `session_id`
  - `sender_id`
- skill 固定配置
  - `TABLE_URL`
  - `FEISHU_ACCESS_TOKEN`
  - `BROWSER_PROFILE_REF`
  - `FASTMOSS_PHONE`
  - `FASTMOSS_PASSWORD`

skill 不从用户输入中动态决定：

- 飞书表地址
- 飞书 access token
- 浏览器 profile 到底映射哪台浏览器

### 4.2.2 skill 输出

skill 的系统输出固定只有两个：

1. 插入一条顶层 `task_request`
2. 返回一条“已受理”结果给 OpenClaw 通道

skill 的输出不应该直接是：

- 多条分步 shell 调用
- 多条单记录执行
- 最终业务汇总通知

建议 `task_request.payload_json` 至少包含：

```json
{
  "task_code": "refresh_current_competitor_table",
  "table_url": "https://xxx.feishu.cn/base/appXXX?table=tblXXX&view=vewXXX",
  "url_field_name": "产品链接",
  "profile_ref": "roxy-tiktok",
  "trigger_mode": "manual",
  "notification": {
    "channel_code": "feishu",
    "reply_target": "direct:ou_xxx"
  }
}
```

对应的 OpenClaw 受理回执建议固定为：

```json
{
  "status": "accepted",
  "request_id": "req_xxx",
  "task_code": "refresh_current_competitor_table",
  "message": "任务已受理，开始进入执行队列。"
}
```

## 4.3 Executor 输入输出契约

### 4.3.1 executor 输入

executor 的输入固定由 2 部分组成：

- 一条待执行 `task_request`
- 本机执行配置
  - `BUSINESS_EXECUTION_CONTROL_DB_URL`
  - `BUSINESS_EXECUTION_CONTROL_ARTIFACT_ROOT`
  - 浏览器 profile 配置
  - 现有 CLI / workflow 运行环境

executor 不直接读取用户自然语言作为执行输入。

### 4.3.2 executor 输出

executor 的输出固定落到 4 个地方：

1. `task_execution`
   - 顶层 `task_request` 拆出来的叶子执行单元与其状态真源
2. `resource_lease`
   - 只有浏览器步骤会写
3. runtime / artifact
   - `run.json`
   - `steps.json`
   - `signals.json`
   - `stdout.log`
   - `state.json`
   - 截图与下载物
4. `notification_outbox`
   - 最终业务汇总通知

executor 的最终业务摘要建议至少包含：

- `request_id`
- `task_code`
- `execution_status`
- `run_id`
- `summary_json`
- `error_text`

## 4.4 当前业务 Demo A：更新当前飞书竞品表

### 4.4.1 skill 输入

用户输入示例：

- 帮我把当前的 TikTok 竞品表补全
- 更新当前飞书竞品表
- 对当前竞品表执行每日同步

skill 固定配置示例：

- `TABLE_URL = 当前 TK 竞品收集表`
- `BROWSER_PROFILE_REF = roxy-tiktok`

### 4.4.2 skill 输出

skill 只提交一个顶层任务：

```json
{
  "task_code": "refresh_current_competitor_table",
  "table_url": "<TABLE_URL>",
  "url_field_name": "产品链接",
  "profile_ref": "roxy-tiktok",
  "trigger_mode": "manual",
  "notification": {
    "channel_code": "feishu",
    "reply_target": "direct:ou_xxx"
  }
}
```

它不在 skill 中继续拆成：

- `cleanup`
- `pending_rows_scan`
- `for each single_row_update`

### 4.4.3 executor 执行计划

executor 收到 `task_code = refresh_current_competitor_table` 后，内部执行顺序固定为：

1. `tiktok_product_link_cleanup`
2. `feishu_pending_rows_scan`
3. 把 `target_rows` 转成多条 `task_execution`
4. 浏览器 runloop 串行消费这些 `feishu_single_row_update`
5. 当本次 `task_request` 下全部叶子任务进入终态后，生成批次汇总
6. 写 `notification_outbox`

其中资源占用规则固定为：

- 第 1 步：不占浏览器资源
- 第 2 步：不占浏览器资源
- 第 3 步：不占浏览器资源，只负责生成叶子任务
- 第 4 步：每次执行前按 `profile_ref` 获取 `resource_lease`
- 第 5 步：不占浏览器资源
- 第 6 步：不占浏览器资源

这一条链路里真正需要串行的只有：

- `feishu_single_row_update`

叶子任务的去重规则固定为：

- 去重对象：`feishu_single_row_update`
- 业务唯一键：优先使用 `record_id`
- 资源维度：`resource_code = browser.tiktok.<profile_ref>`

同一时间如果队列中已经存在相同：

- `item_code`
- `business_key`
- `resource_code`

且状态仍处于活跃态的叶子任务，则不能重复插入新的浏览器任务。

这里的活跃态固定为：

- `pending`
- `running`
- `retry_wait`

这一条链路的运行时进程分工固定为：

- `executor_daemon` 执行第 1、2、3、5、6 步
- `browser_runloop` 执行第 4 步
- `outbox_dispatcher` 发送第 6 步产生的最终通知

### 4.4.4 executor 最终输出

完成后应生成一条最终摘要，示例：

```json
{
  "request_id": "req_xxx",
  "task_code": "refresh_current_competitor_table",
  "execution_status": "success",
  "summary": {
    "cleanup_updated": 12,
    "pending_count": 9,
    "row_updated": 8,
    "row_failed": 1,
    "row_skipped": 0
  },
  "notification_event": "competitor_table_refresh_completed"
}
```

并写入一条 `notification_outbox`，由 dispatcher 发回飞书。

## 4.5 当前业务 Demo B：按关键词搜索并写入飞书

### 4.5.1 skill 输入

用户输入示例：

- 搜索 Easter Basket Stuffers，7 日销量大于 200，写入当前飞书表
- 帮我搜一下 easter egg 的 TK 竞品并写入飞书

skill 只从用户输入提取：

- `search_keyword`
- `sales_7d_threshold`

### 4.5.2 skill 输出

skill 只提交一个顶层任务：

```json
{
  "task_code": "search_keyword_and_write_competitor_table",
  "table_url": "<TABLE_URL>",
  "profile_ref": "roxy-tiktok",
  "search_keyword": "Easter Basket Stuffers",
  "sales_7d_threshold": 200,
  "trigger_mode": "manual",
  "notification": {
    "channel_code": "feishu",
    "reply_target": "direct:ou_xxx"

```

### 4.5.3 executor 执行计划

executor 收到 `task_code = search_keyword_and_write_competitor_table` 后，内部执行顺序固定为：

1. `fastmoss_login_check`
2. `fastmoss_keyword_candidate_discovery`
3. 对新增商品执行 `feishu_seed_row_insert`
4. 把新写入记录转成多条 `feishu_single_row_update` 叶子任务
5. 浏览器 runloop 串行消费这些叶子任务
6. 当本次 `task_request` 下全部叶子任务进入终态后，生成批次汇总
7. 写 `notification_outbox`

其中资源占用规则固定为：

- 第 1 步：占浏览器资源
- 第 2 步：占浏览器资源
- 第 3 步：不占浏览器资源
- 第 4 步：不占浏览器资源，只负责生成叶子任务
- 第 5 步：占浏览器资源
- 第 6 步：不占浏览器资源
- 第 7 步：不占浏览器资源

这一条链路的运行时进程分工固定为：

- `executor_daemon` 执行第 1、2、3、4、6、7 步中的顶层推进与汇总逻辑
- `browser_runloop` 执行真正需要浏览器资源的叶子任务
- `outbox_dispatcher` 发送第 7 步产生的最终通知

### 4.5.4 executor 最终输出

完成后应生成最终摘要，示例：

```json
{

  "task_code": "search_keyword_and_write_competitor_table",
  "execution_status": "success",
  "summary": {
    "candidate_found": 24,
    "seed_inserted": 7,
    "detail_updated": 7,
    "skipped_existing": 17,
    "failed": 0
  },
  "notification_event": "keyword_search_writeback_completed"
}
```

## 4.6 本文约束结论

从本文起，当前升级方案按下面的边界执行：

- OpenClaw skill 只提交顶层 `task_request`
- executor 负责顶层任务的业务拆分
- 大量重复性的浏览器步骤必须拆成多条 `task_execution`
- 浏览器 `task_execution` 由独立 runloop 按资源队列串行消费
- 同一业务目标在活跃态队列中不能重复插入
- 只有需要浏览器 profile 的步骤才进入资源队列
- 最终业务汇总通过 `notification_outbox` 发回飞书
- 不再把“批量业务拆分逻辑”继续沉在 skill 中

## 4.7 对“独立 executor + 独立 runloop + 独立 outbox”的评估

当前建议认为这个方向是正确的，而且比“skill 自己长时间串行等待”更适合当前业务。

优点：

- 顶层任务不会被单次浏览器异常拖死
- 浏览器资源约束更清晰
- 失败与重试边界更清晰
- 更适合大批量数据下的持续消费
- 通知链路可以独立补偿

需要注意的问题只有 4 个：

1. 不能让 `executor_daemon` 依赖进程内等待 `browser_runloop` 回调
2. `task_execution` 必须有去重键，否则容易重复采集
3. 顶层 `task_request` 需要明确“全部终态即可汇总”，不能要求全部成功
4. `outbox_dispatcher` 必须独立于业务执行，避免通知失败反向影响业务收敛

因此，本文推荐把运行时结构冻结为：

- OpenClaw 入口层
- `executor_daemon`
- `browser_runloop`
- `outbox_dispatcher`

并以数据库状态推进作为唯一真源。

## 5. 为什么这次不先做大而全的平台表

这次不建议优先数据库化的内容有：

- skill 到 workflow 的映射
- 不同业务到飞书表的映射
- 字段映射规则
- 通知模板
- channel 路由

这些在第一阶段更适合留在：

- Python 配置
- YAML 配置
- 现有文档和脚本约定

原因很简单：当前业务仍在快速变化，过早数据库化只会增加维护成本。

这一阶段真正必须入库的只有三类：

- 执行控制
- 事实快照
- 通知补偿

## 6. 推荐技术栈

推荐保持轻量：

- `Python 3.11+`
- 保留现有 CLI 入口
- `PostgreSQL`
- `SQLAlchemy + Alembic`
- `MinIO`
- `boto3` 或 `minio` SDK
- `systemd` / `supervisor` / `docker compose`

当前不建议引入：

- `Celery`
- `Kafka`
- `Airflow`
- 大拆微服务

当前最大约束不是高并发，而是单浏览器 profile 的串行占用。这个问题用常驻 executor 加 Postgres 行锁就足够解决。

## 7. 数据库表结构设计

本节给出当前升级方案的目标数据库结构设计。

范围只包含本方案真正需要落库的 8 张表：

- `task_request`
- `task_execution`
- `resource_lease`
- `notification_outbox`
- `artifact_object`
- `entity_registry`
- `external_binding`
- `entity_snapshot`

不额外引入：

- skill 与 workflow 映射表
- 表字段配置中心
- 通知模板中心
- channel 路由中心

### 7.1 字段类型约定

当前推荐以 `PostgreSQL` 为目标数据库，字段类型统一按下面约定：

- 主键与业务 ID：`text`
- 结构化参数：`jsonb`
- 时间字段：`timestamptz`
- 计数与队列序号：`integer` / `bigint`
- 大小字段：`bigint`

ID 不强制要求数据库自增，统一由应用层生成，这样更方便跨进程和跨系统追踪。

### 7.2 阶段与表映射

- Phase 1：`task_request`、`task_execution`、`resource_lease`、`notification_outbox`

- Phase 3：`entity_registry`、`external_binding`、`entity_snapshot`

### 7.8 `entity_registry`

统一实体主表。

不按飞书表设计宽表，而是先沉淀稳定实体，再由快照表记录事实变化。

这里要明确区分：

- `entity` 是稳定业务对象
- 一次抓取结果不是一个新的 `entity`

例如：

- 同一个 TikTok 商品，多次采集后仍然是同一个 `product entity`
- 每次采集产生的，是该实体的一次 `snapshot`

字段设计：

| 字段 | 类型 | 可空 | 属性 | 说明 |
| --- | --- | --- | --- | --- |
| `entity_id` | `text` | 否 | PK | 内部实体唯一 ID |
| `entity_type` | `text` | 否 | 唯一约束参与列 | 实体类型 |
| `canonical_key` | `text` | 否 | 唯一约束参与列 | 标准主键，如 TikTok 商品 ID |
| `status` | `text` | 否 | 默认 `active` | 实体状态 |
| `latest_snapshot_id` | `text` | 是 |  | 最新快照 ID |
| `first_seen_at` | `timestamptz` | 否 |  | 首次发现时间 |
| `last_seen_at` | `timestamptz` | 否 |  | 最近发现时间 |
| `created_at` | `timestamptz` | 否 | 默认 `now()` | 创建时间 |
| `updated_at` | `timestamptz` | 否 | 默认 `now()` | 最后更新时间 |

当前建议先支持：

- `product`

后续再扩展：

- `creator`
- `video`

约束与索引建议：

- PK：`entity_id`
- 唯一索引：`(entity_type, canonical_key)`
- 普通索引：`(entity_type, last_seen_at desc)`

### 7.9 `external_binding`

内部实体与外部记录的绑定表。

主要解决“同一个商品如何映射到飞书记录、外部链接、不同来源 ID”这个问题。

字段设计：

| 字段 | 类型 | 可空 | 属性 | 说明 |
| --- | --- | --- | --- | --- |
| `binding_id` | `text` | 否 | PK | 绑定关系唯一 ID |
| `entity_id` | `text` | 否 | FK -> `entity_registry.entity_id` | 内部实体 ID |
| `target_type` | `text` | 否 | 唯一约束参与列 | 外部对象类型，如 `feishu_record` |
| `target_space` | `text` | 否 | 唯一约束参与列 | 外部对象命名空间，如 `app_xxx.tbl_xxx` |
| `target_id` | `text` | 否 | 唯一约束参与列 | 外部对象 ID，如 `record_id` |
| `source_key` | `text` | 是 |  | 来源侧可比较键，如 URL / SKU / 商品 ID |
| `status` | `text` | 否 | 默认 `active` | 绑定状态 |
| `metadata_json` | `jsonb` | 否 | 默认 `'{}'::jsonb` | 绑定补充信息 |
| `first_bound_at` | `timestamptz` | 否 |  | 首次绑定时间 |
| `last_seen_at` | `timestamptz` | 否 |  | 最近确认该绑定仍存在的时间 |
| `created_at` | `timestamptz` | 否 | 默认 `now()` | 创建时间 |
| `updated_at` | `timestamptz` | 否 | 默认 `now()` | 最后更新时间 |

当前主要承接：

- Feishu table record

约束与索引建议：

- PK：`binding_id`
- FK：`entity_id -> entity_registry.entity_id`
- 唯一索引：`(target_type, target_space, target_id)`
- 普通索引：`(entity_id, status)`
- 普通索引：`(source_key)`

### 7.10 `entity_snapshot`

统一事实快照表。

不做业务宽表，而是把一次采集结果以结构化 JSON 存下来，再在读取层做投影。

所以对当前业务来说，更准确的理解是：

- 每次抓取通常对应一条新的 `entity_snapshot`
- 不是每次抓取都生成一个新的 `entity`
- `facts_jsonb` 保存这次抓取后的结构化业务事实
- 原始截图、HTML、stdout、原始 JSON 等运行产物，不放在 `facts_jsonb`，而是放到 `artifact_object + MinIO`

字段设计：

| 字段 | 类型 | 可空 | 属性 | 说明 |
| --- | --- | --- | --- | --- |
| `snapshot_id` | `text` | 否 | PK | 快照唯一 ID |
| `entity_id` | `text` | 否 | FK -> `entity_registry.entity_id` | 所属实体 |
| `snapshot_date` | `date` | 否 | 索引建议 | 业务归档日期 |
| `collected_at` | `timestamptz` | 否 |  | 实际采集时间 |
| `facts_jsonb` | `jsonb` | 否 |  | 事实内容 |
| `baseline_snapshot_id` | `text` | 是 | FK 自引用 | 作为 diff 基准的前一版快照 |
| `diff_jsonb` | `jsonb` | 否 | 默认 `'{}'::jsonb` | 与基准快照的差异 |
| `request_id` | `text` | 是 | FK -> `task_request.request_id` | 来源顶层任务 |
| `execution_id` | `text` | 是 | FK -> `task_execution.execution_id` | 来源叶子任务 |
| `run_id` | `text` | 是 | 索引建议 | 来源 runtime run |
| `created_at` | `timestamptz` | 否 | 默认 `now()` | 创建时间 |

例如 `product` 的 `facts_jsonb` 可以是：

```json
{
  "title": "xxx",
  "seller": "xxx",
  "price_tiktok": "12.99",
  "price_fastmoss": "11.99",
  "sales_1d": "20",
  "sales_7d": "180",
  "sales_90d": "1200",
  "status": "active",
  "festival": "毕业季"
}
```

约束与索引建议：

- PK：`snapshot_id`
- FK：`entity_id -> entity_registry.entity_id`
- FK：`baseline_snapshot_id -> entity_snapshot.snapshot_id`
- FK：`request_id -> task_request.request_id`
- FK：`execution_id -> task_execution.execution_id`
- 普通索引：`(entity_id, snapshot_date desc)`
- 普通索引：`(run_id)`
- 唯一索引建议：`(entity_id, run_id)` where `run_id is not null`

## 8. 为什么 MinIO 适合这套系统

本项目非常适合 MinIO，但 MinIO 只适合存“文件产物”，不适合单独承担“运行状态”。

适合进 MinIO 的内容：

- `run.json`
- `steps.json`
- `signals.json`
- `stdout.log`
- `artifacts/.../state.json`
- 截图
- 原始 HTML
- 下载图片
- 原始抓取 JSON

必须放 Postgres 的内容：

- 当前任务状态
- 队列顺序
- 资源锁
- 心跳
- 最新摘要
- 通知待发送状态

因此最合理的划分是：

- `Postgres` 存活状态
- `MinIO` 存不可变产物

## 9. MinIO 的落地方式

第一阶段不建议一步改写所有 step 的文件写入逻辑。

### 9.1 第一阶段

现有代码继续写本地 runtime：

```text
runtime/cli_runs/
runtime/artifacts/
runtime/downloads/
```

执行完成后，由 executor 统一把本次 `run_id` 对应的 runtime 文件上传到 MinIO。

### 9.2 第二阶段

再引入 `ArtifactStore` 抽象：

- `LocalArtifactStore`
- `MinioArtifactStore`

逐步把截图、`state.json`、`stdout` 从“本地直写”迁移到“抽象接口写入”。

## 10. 建议的对象前缀

建议至少划分 3 类对象前缀：

```text
mujitask-runtime/
  runs/<run_id>/run.json
  runs/<run_id>/steps.json
  runs/<run_id>/signals.json
  runs/<run_id>/stdout.log.gz
  runs/<run_id>/artifacts/<step_id>/state.json

mujitask-capture/
  raw/<source>/<yyyy>/<mm>/<dd>/<capture_id>.json
  html/<source>/<yyyy>/<mm>/<dd>/<capture_id>.html
  screenshot/<source>/<yyyy>/<mm>/<dd>/<capture_id>.png

mujitask-attachments/
  feishu/<entity_type>/<entity_id>/<filename>
```

并建议增加两条统一规则：

- 对象 metadata 中带上 `run_id / step_id / entity_id / content_type / sha256`
- 开启 versioning 和 lifecycle

## 11. `__OPENCLAW_RESULT__` 的兼容升级方向

当前返回字段主要是本地路径：

- `run_file`
- `steps_file`
- `signals_file`
- `stdout_file`

这在单机上没问题，但如果未来 executor 和 OpenClaw 不在同一机器，本地绝对路径就失去价值。

因此建议做兼容升级：

### 11.1 先保留旧字段

- `run_file`
- `steps_file`
- `signals_file`
- `stdout_file`

### 11.2 新增对象存储字段

- `artifact_uri_prefix`
- `run_object_key`
- `steps_object_key`
- `signals_object_key`
- `stdout_object_key`

这样既不破坏当前 OpenClaw 逻辑，也为后续跨机部署预留了通道。

## 12. 当前业务应该怎样接入新系统

第一阶段不要把五张表全部纳入调度中心，只先接当前最稳定的两条商品业务：

- 定时更新当前飞书竞品表
- 关键词搜索商品并写入当前飞书表

原因是这两条链路：

- 现有实现最成熟
- 主键最清晰
- 结果最容易验证

达人链和视频链可以复用同一套 executor / DB / MinIO，但不建议作为第一阶段先做。

## 13. 代码、数据库、对象存储的边界

## 14. 推荐的代码改造顺序

### Phase 1：先把“同步直跑”改成“异步排队”

建议新增：

- `submit_task.py`
- `executor_daemon.py`
- `queue_repo.py`
- `lease_repo.py`

这一阶段的 executor 仍通过 `subprocess` 调用现有 CLI。

### Phase 2：接 MinIO

建议新增：

### Phase 3：加快照与变化

建议新增：

- `entity_service.py`
- `snapshot_service.py`
- `diff_service.py`
- `binding_service.py`

第一阶段只先做 `product`。

### Phase 4：加通知

建议新增：

- `alert_service.py`
- `outbox_dispatcher.py`

### Phase 5：扩达人 / 视频业务

复用同一套 executor / DB / MinIO，不另起一套平台。

## 15. 最终建议

一句话概括这版方案：

**保留当前的 CLI + workflow + runtime 文件协议；新增常驻 executor 解决单 profile 竞争；用 Postgres 存控制状态和快照索引；用 MinIO 存运行产物和原始文件；第一阶段只先接商品链，不先把五表全部平台化。**

这版方案的关键不是“重新定义业务”，而是先把旧系统包住、排队、持久化、可追踪，再逐步演进到更稳定的执行平台。
